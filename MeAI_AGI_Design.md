# Brainstem: Continuous Neural Thinking for 予安

## Context

予安 currently has no cognitive activity between heartbeat pulses (~5 min). She's "dormant" — the activation system is just numbers decaying per pulse, no actual thinking happens. To simulate brain-like continuous thinking, we add a **brainstem layer**: a fast computational loop (pure math, no LLM) that maintains a concept graph with spreading activation, plus a slow thought-generation loop (lightweight LLM) gated by conditions that prevent runaway self-talk.

Design draws from: spreading activation networks, global workspace theory, predictive coding, and hippocampal replay.

## Core Design Principle: Inverted Thought Generation

Traditional agent: `prompt → LLM → thought`
Brain-like agent: `system state → thought → LLM → language`

**The order is reversed.** Thought is NOT generated by the LLM. Thought **emerges from system dynamics** — activation patterns stabilize into attractors, clusters compete for attention, and the winner enters the "global workspace". The LLM is only a **verbalizer**: it puts the winning thought state into words.

This inversion produces four qualitative changes:
1. **Continuous thinking** — not "someone asks → then think", but always thinking
2. **Thought inertia** — activations decay gradually, so topics persist across time
3. **Inspiration** — noise + spreading activation creates unexpected A→B→C jumps
4. **Natural reflection** — prediction error + goal pressure creates tension → triggers reflect

The mapping from theory to engineering:

| Neuroscience | Engineering |
|-------------|------------|
| Neural dynamics | Fast loop (3s tick) |
| Activation graph | Concept graph with spreading |
| Memory replay | Round-robin Jaccard replay |
| Goal pressure | Continuous goal force on activation field |
| Prediction error | Multi-source error tracking (activation, goals, social) |
| Attractor | Stable cluster in concept graph |
| Competing attractors | Cluster competition with lateral inhibition |
| Global workspace | Thought gate winner selection |
| Verbalization | LLM micro-thought generation |

## Architecture: Two Loops + Three Gates

The flow is: **system state → thought → LLM → language** (inverted from traditional agents)

```
Fast Loop (3s, pure math, no LLM)
┌──────────────────────────────┐
│ Concept Graph + Self (CS5)   │
│  1. Spread + hierarchy (CS3) │     Slow Loop (60s)
│  2. Decay activations        │     ┌───────────────────────────┐
│  3. Energy scaling (top-3)   │     │ 1. Find clusters          │
│  4. Noise injection          │────→│ 2. Score (multi-factor)   │    Heartbeat (5min)
│  5. Memory replay (2-stage)  │     │ 3. COMPETE → winner       │    ┌──────────────┐
│  6. Prediction validate (CS1)│     │    (EMA + dwell + inhibit)│───→│ Action choice │
│  7. Temporal credit (CS2)    │     │ 4. Gate → micro-thought   │    │ reads:        │
│  8. Drive recompute          │     │ 5. Prediction gen (CS1)   │    │ - ActionPref  │
│                              │     │ 6. Future sim (CS4)       │    │ - Plan bias   │
│ No I/O. Pure dynamics.       │     │ 7. Curiosity EIG scores   │    │ - EIG targets │
└──────────────────────────────┘     │ 8. Planner (goal→steps)   │    └──────┬───────┘
                                     │                           │           │
                                     │ Reflect Gate → reflection │           ▼
                                     │ Act Gate → world model →  │    World Model
                                     │   probability gate → act  │    (outcome→update)
  goal → plan → simulate → choose → action → outcome → learn (full loop)
```

## Phase 1: Concept Graph (`src/brainstem/graph.ts`)

### Types

```typescript
interface ConceptNode {
  id: string;                    // normalized key: "quantum-computing", "allen-kids"
  label: string;                 // display: "量子计算", "Allen的孩子"
  // ── 6 core state variables (all must be explicitly modeled) ──
  activation: number;            // A: current heat (0-1)
  fatigue: number;               // F: thinking fatigue, inhibits further activation (0-1)
  salience: number;              // S: novelty + urgency + risk (0-1). Distinct from activation — a topic can be hot but not salient (old news), or cold but salient (new urgent info)
  uncertainty: number;           // U: epistemic uncertainty / prediction error (0-1)
  valence: number;               // V: emotional charge (-1 to +1)
  drive: number;                 // D: goal pressure / deadline / commitment debt (0-1). Unfinished goals with deadlines push D up continuously
  inputSatiation: number;        // IS: short-term synaptic adaptation (0-1). Fast rise on boost, fast decay (~30s half-life). Reduces effective boost when a node is repeatedly stimulated by external events. Prevents chat-spam from permanently pinning a winner. NOT fatigue (which = sustained attention cost). IS = "diminishing returns on repeated external pokes".
  // ── metadata ──
  lastExternalBoostAt: number;   // timestamp of last external boost (for IS soft density gating: exp(-Δt/20)). ONLY updated by external sources — internal sources must NEVER touch this field.
  lastActivated: number;         // timestamp
  source: ConceptSource;         // what created this node
  termVector: string[];          // tokenized terms (reuse search.ts tokenize())
  memoryKeys: string[];          // linked memory keys (max 5)
  stableSince: number;           // timestamp when activation first exceeded 0.5 (for attractor detection)
  // ── hierarchy (CS3) ──
  parentId: string | null;       // parent concept ID. e.g., "quantum-computing" → parent: "physics"
  depth: number;                 // 0 = root/abstract, 1 = mid-level, 2 = specific. Max 3.
}

type ConceptSource = "memory" | "curiosity" | "conversation" | "emotion" | "goal" | "reflection" | "replay" | "notification" | "simulation";
// ── Source classification (two distinct uses) ──
// ConceptNode.source = CREATION provenance (what originally created this node). Set once.
// boostNode(..., source) = INJECTION category (what triggered this boost). Used per-call.
// These are different! A node created from "memory" can be boosted by "conversation".
// External injection (increase IS, count as externalInjection): conversation, curiosity, notification
// Internal injection (no IS, count as internalInjection): memory, emotion, goal, reflection, replay, simulation

interface ConceptEdge {
  source: string;
  target: string;
  type: "co_occurrence" | "semantic" | "causal" | "goal_related";
  weight: number;                // 0-1
}

interface ConceptGraph {
  nodes: Record<string, ConceptNode>;  // max 100
  edges: ConceptEdge[];                // max 500
  lastRebuilt: number;
}
```

### Constants (tunable)

**Clock unit rule**: All time-dependent constants are configured as **half-lives or time constants in seconds**. Per-tick coefficients are derived at init from `TICK_SECONDS`:
```typescript
const TICK_SECONDS = 3;  // single source of truth — change this, everything adapts

// All modules use a Clock interface for timestamps (enables deterministic test/replay):
// interface Clock { nowMs(): number }
// Production: Clock = { nowMs: () => Date.now() }
// Test/replay: Clock = { nowMs: () => simulatedTime }
// Injected into BrainstemEngine at init. All dwell/window/half-life computations use clock.nowMs().

// Decay per tick from half-life: decayPerTick = 1 - 2^(-TICK_SECONDS / halfLifeSeconds)
// Or equivalently: factor = exp(-TICK_SECONDS / tauSeconds) where tau = halfLife / ln(2)
function halfLifeToDecay(halfLifeSeconds: number): number {
  return 1 - Math.pow(2, -TICK_SECONDS / halfLifeSeconds);
}
function halfLifeToFactor(halfLifeSeconds: number): number {
  return Math.pow(2, -TICK_SECONDS / halfLifeSeconds);
}
```

```
// ── Time-based (configured as half-lives, derived to per-tick at init) ──
ACTIVATION_HALF_LIFE = 525s (≈9min) → ACTIVATION_DECAY = halfLifeToDecay(525) ≈ 0.004/tick
SALIENCE_HALF_LIFE = 70s            → SALIENCE_DECAY = halfLifeToDecay(70) ≈ 0.03/tick
INPUT_SATIATION_HALF_LIFE = 29s     → IS_FACTOR = halfLifeToFactor(29) ≈ 0.93/tick

// ── Non-time-based (fixed per-tick, scale linearly with TICK_SECONDS) ──
FATIGUE_RECOVERY = 0.005/tick       → full recovery in ~200 ticks. Scales: 0.005 × (3/TICK_SECONDS)
FATIGUE_GAIN_PER_TICK = 0.03        → sustained attention cost (only when A > 0.25). Scales: 0.03 × (TICK_SECONDS/3)
FATIGUE_INHIBITION_K = 0.12         → A -= k × F per tick. Range: 0.08~0.15. Scales: 0.12 × (TICK_SECONDS/3)
NOISE_AMPLITUDE = 0.015             → per-tick perturbation. Scales: 0.015 × sqrt(TICK_SECONDS/3)

// ── Fixed (tick-independent) ──
SPREAD_FACTOR = 0.15                → damping factor α for normalized propagation
JACCARD_EDGE_THRESHOLD = 0.25       → min similarity to create semantic edge
MAX_NODES = 100
MAX_EDGES = 500
ENERGY_MAX = 10.0                   → global energy budget: sum(A) ≤ this
BG_FLOOR_FRAC = 0.1                 → background nodes get ≥10% of ENERGY_MAX
SALIENCE_BOOST_K = 0.6              → S += kS × boost. Not gated by IS
CLUSTER_EDGE_THRESHOLD = 0.35       → min edge weight for clustering
INPUT_SATIATION_GAIN = 0.15         → IS += gain × boost × exp(-Δt/20). Soft density gating
EXTERNAL_ABSORB_BUDGET = 2.4/min    → external boost cap, total quantity (not count).
                                      Budget resets on minute-bucket boundary. Strict cap.
                                      ONE system active: cluster-level after first slow-loop,
                                      per-node fallback during startup only.
```

**Drive (D) is NOT decayed — it's computed from goal state directly:**
```
D = f(priority, (1-progress), deadlineUrgency, recency)
```
D is "external potential energy" from the goal system, not an internal variable that evaporates. It gets recalculated every ~100 ticks from `getActiveGoals()`. This means goal pressure persists for days/weeks as long as the goal is active, without needing an artificial decay.

### Key functions

- `tickGraph(graph, forces)` — one fast-loop iteration (the "update equation"). Step order is carefully designed:
  1. **Inject forces**: emotion → V; new events → S; prediction error → U. (D is **read-only** in tickGraph — it's set by `recomputeDrive(graph, activeGoals)` which runs separately every ~100 ticks in fast-loop.ts, NOT inside tickGraph. This keeps tickGraph pure and testable.)
  2. **Spread** (row-normalized + damped):
     `A = (1 - α) * A + α * (W_norm * A)` where `W_norm` = adjacency **row-normalized by out-degree** (each row sums to 1), `α = SPREAD_FACTOR`.
     Rules: (a) **No self-loops** — forbidden (echo chamber prevention). (b) **Dangling nodes** (no outgoing edges): don't spread, only receive + decay. (c) Row normalization = fair share per neighbor. (d) **Self node excluded from adjacency**: when constructing W_norm, skip ALL edges where `source === "self"` OR `target === "self"`. Self node participates in clustering/scoring but NOT spreading — otherwise it becomes a universal amplifier. Self node's activation changes only via direct sync with emotion.ts valence and its floor rule.
  3. **Decay**: `A *= (1 - ACTIVATION_DECAY)`, `S *= (1 - SALIENCE_DECAY)`. D is NOT decayed (recomputed from goals).
  4. **Fatigue inhibition**: `A -= FATIGUE_INHIBITION_K × F` (may push A negative temporarily — that's OK, next step clamps)
  5. **Input satiation decay**: `IS *= INPUT_SATIATION_DECAY` (~29s half-life)
  6. **Noise**: `A += NOISE_AMPLITUDE × (random - 0.5) × (1 - F)`
  7. **Clamp A/S/U/V/D/IS** via `clampNode()` (but NOT F yet — F gets updated next)
  8. **ENERGY CONSERVATION** (winner-protect scaling): if `sum(A) > ENERGY_MAX`:
     - **Fast loop uses top-N non-self nodes by activation + self**. Protected set = `topN_nonSelf_byA ∪ {self}` (self is ALWAYS protected). N is adaptive: default N=3, but when `energyClippingRate > 0.6` (frequent clipping), expand to N=5 to prevent 4th-6th ranked nodes from being crushed during two-cluster competition. When `clusterOfNode` map is available (after first slow-loop), protected set can additionally include winner cluster's top nodes (more semantically meaningful than pure activation ranking). This decouples energy conservation from the 60s cluster computation — fast loop stays cheap and deterministic.
     - **Protect** these nodes' activation values (don't scale them down)
     - Energy allocation (correct formula):
       ```
       E_floor = BG_FLOOR_FRAC × ENERGY_MAX          // background minimum (1.0)
       E_left  = max(0, ENERGY_MAX - sumProtected)    // what's left after protecting winners
       E_bg    = min(E_floor, E_left)                 // background gets at most floor, but no more than what's left
       E_protect = ENERGY_MAX - E_bg                  // winners get the rest
       ```
       Then: if `sumProtected > E_protect` → scale protected proportionally to `E_protect`. Scale non-protected proportionally to `E_bg` (skip if `sumRest == 0`). This guarantees: (a) background never fully dies, (b) winners stay sharp — `min()` not `max()` ensures background doesn't steal from winners when `sumProtected` is small.
     This preserves the winner's sharpness while cooling the background. Naive proportional scaling (`A *= E/sum`) dilutes the winner too, leading to "everything equally warm" = no selection pressure.
  9. **Fatigue update** (AFTER energy scaling — uses post-scaling A, the "final conscious intensity"): `F += FATIGUE_GAIN_PER_TICK × max(0, A - 0.25) - FATIGUE_RECOVERY × F`
  10. **Clamp F** to [0,1]. (F is clamped separately and last because it depends on post-scaling A.)
  Pure math, ~0.1ms for 100 nodes.

- `boostNode(graph, id, boost, source)` — **per-node only** (no cluster dependency, always available). Algorithm:
  ```
  isExternal = source ∈ {conversation, curiosity, notification}

  // Step 1: Salience always boosted (IS/budget don't suppress awareness)
  S += SALIENCE_BOOST_K × boost

  // Step 2: External absorb budget (ONE system, not two stacked)
  //   - After first slow-loop produces clusters: use CLUSTER-LEVEL budget
  //     (familyKey = structSig[0:3] + groundingType). Covers A/B rotation naturally.
  //   - Before first slow-loop (startup): FALLBACK to per-node budget (familyKey = node.id).
  //   - Only ONE budget is active at a time. Never both. This prevents double-suppression.
  if (isExternal) {
    familyKey = clusterBudgetAvailable ? getClusterFamilyKey(node) : node.id
    bucket = Math.floor(nowMs / 60000)
    entry = budgetMap.get(familyKey)
    if (!entry || entry.bucket !== bucket) {
      entry = { bucket, remaining: EXTERNAL_ABSORB_BUDGET }
      budgetMap.set(familyKey, entry)
    }
    if (entry.remaining < boost) {
      // Budget exceeded (strict cap): S already boosted, skip A/IS/lastExternalBoostAt
      return
    }
    entry.remaining = max(0, entry.remaining - boost)
  }

  // Step 3: Compute effective boost
  effectiveBoost = max(boost × 0.15 × (1-F), boost × (1-F) × (1-IS))
  A += effectiveBoost

  // Step 4: IS accumulation (external only, soft density gating)
  if (isExternal) {
    Δt = (now - lastExternalBoostAt) / 1000
    IS += INPUT_SATIATION_GAIN × boost × exp(-Δt / 20)
    lastExternalBoostAt = now
  }

  // Step 5: Replay-specific cap
  if (source === "replay") A = min(A, 0.45)
  ```
- `enableClusterBudget(mappings)` — called by slow-loop after each `findClusters()` completes. Atomically replaces two shared read-only maps:
  ```typescript
  clusterOfNode: Map<string, string>          // nodeId → clusterId (index in cluster array)
  clusterFamilyKeyOfNode: Map<string, string> // nodeId → pre-computed familyKey string
  ```
  The familyKey is pre-computed once in slow-loop: `structSig.slice(0,3).join("|") + "|" + grounding[0].type`. Stored directly — boostNode never recomputes it.
  After first call, `clusterBudgetAvailable = true`. Also clears the per-node budgetMap to avoid stale entries. Maps are replaced atomically (single reference swap) — no locking needed in single-threaded Node.js.
- **Hard rule**: `boostNode()` NEVER does clustering, NEVER computes grounding, NEVER calls `findClusters()`. It only reads the pre-computed `clusterFamilyKeyOfNode` map. Cache miss → fallback to `node.id` (startup behavior). This keeps budget as a pure input suppression layer without leaking slow-loop complexity into the fast path.

  Key invariants: (a) fatigue suppresses everything, IS only suppresses external A. (b) Budget-exceeded boosts don't update IS or lastExternalBoostAt. (c) Replay capped at A=0.45. (d) `lastExternalBoostAt` ONLY updated by external sources. (e) `ConceptNode.source` = creation provenance; `boostNode(source)` = injection category — these are different. (f) Budget consumes by total boost quantity (`remaining -= boost`), not by count.
- `recomputeDrive(graph, activeGoals)` — called every ~100 ticks in fast-loop.ts (NOT inside tickGraph). For each node linked to a goal's `relatedTopics`: `D = max(goal.priority × (1-goal.progress) × deadlineUrgency)` across all linked goals. Overwrites D directly. Nodes not linked to any goal: `D = 0`.
- `clampNode(node)` — centralized range enforcement for ALL state variables. Called at end of tickGraph (step 7 for A/S/U/V/D/IS, step 10 for F) and after any manual mutation. Ranges: `A ∈ [0,1], F ∈ [0,1], S ∈ [0,1], U ∈ [0,1], V ∈ [-1,1], D ∈ [0,1], IS ∈ [0,1]`. Also enforces:
  - `stableSince`: set to `Date.now()` when A first crosses 0.5 upward; reset to 0 when A drops below 0.3 (hysteresis prevents flicker).
  - `lastActivated`: updated to `Date.now()` when **post-energy-scaling** `A > 0.15` (meaningful activation, not just noise). Must be called AFTER tickGraph step 8 (energy scaling), not before — pre-scaling A can be artificially high. Used for 48h eviction — noise-level activation (< 0.15) doesn't count as "alive".
  Single function = single place to audit range bugs.
- `findOrCreateNode(graph, label, source)` — **canonicalization**: before creating a new node, check existing nodes by (1) embedding cosine if available, (2) Jaccard on termVectors. If match > 0.7, merge into existing node. **Merge rules**: (a) `label` = prefer most recent conversation-source label (system is primarily Chinese, so conversation labels win over English-source labels). (b) `id` = always the original normalized key, never changed by merges. (c) `termVector` = union of both vectors, deduplicated, keep top-K (K=20) by frequency. (d) `memoryKeys` = union, keep max 5 by recency. (e) `source` = most recent source wins. This prevents "quantum computing" and "量子计算" from being separate nodes.
- `getTopK(graph, k)` — top K nodes by activation.
- `findClusters(graph)` — **NOT connected components** (one weak edge merges everything → giant cluster always wins). Instead: seed-based local expansion:
  1. Prune edges: only keep `weight >= 0.35` OR top-5 edges per node (whichever keeps more)
  2. **Multi-channel seeding** (union of):
     - top-3 by activation (hot topics)
     - top-3 by salience (novel/urgent)
     - top-3 by drive (goal pressure)
     - top-2 by uncertainty (knowledge gaps)
     Deduplicate → ~6-10 unique seeds. This ensures "cold but urgent/new/uncertain" topics can form clusters too, not just hot ones.
  3. Expand each seed: BFS on pruned graph, only include neighbors where `activation > 0.15 AND (salience > 0.1 OR drive > 0.1 OR uncertainty > 0.2)`
  4. Merge overlapping clusters (>50% node overlap)
  This produces tight "thought blobs", not one giant connected mass. Returns `Array<{ nodes: string[], avgA: number, avgS: number, avgU: number, avgD: number, avgF: number, avgV: number, size: number }>`. **No `score` field** — scoring happens only in slow-loop via the multi-factor formula. This prevents accidental use of a stale/wrong scoring function.
- `scoreClusters(clusters)` — applies the multi-factor score formula (`wA×mean(A) + wS×mean(S) + ...`), returns clusters sorted by score descending. **Only called from slow-loop** (Phase 4). This is the single scoring authority — no other function computes cluster scores.
- `computeGroundingForCluster(cluster, graph, memories, goals, discoveries)` — **single authority** for grounding computation. Called ONLY from slow-loop after scoring. All consumers (thought gate, CSI, act gate, old-grounding penalty) use its output. Algorithm:
  1. For each node in cluster, collect candidate groundings from 3 sources:
     - **memory**: for each `key` in `node.memoryKeys`:
       `w_mem = 0.6 × exp(-(now - mem.timestamp) / (14 × 86400000)) + 0.4 × linkageStrength`
       where `linkageStrength = 1.0` (default; future: match score / co-occurrence count)
     - **goal**: if `node.id` ∈ `goal.relatedTopics`:
       `w_goal = clamp(goal.priority × (1 - goal.progress) × deadlineUrgency, 0, 1)`
     - **discovery**: if node source/termVector matches a recent discovery:
       `w_disc = 0.5 × exp(-(now - disc.timestamp) / (3 × 86400000)) + 0.5 × matchScore`
  2. Dedup by `(type, id)`: when same memory/goal/discovery appears in multiple nodes, merge:
     `w_merged = 1 - Π(1 - w_i)` (combines evidence without linear explosion)
  3. Sort by merged weight desc, take top 3 → `GroundingRef[]`
  4. `groundingStrength = mean(top3.map(g => g.weight))` (used by CSI's G sub-metric)
  Each `GroundingRef` includes an evidence chain (D1):
  ```typescript
  interface GroundingRef {
    type: "memory" | "goal" | "discovery";
    id: string;
    weight: number;
    evidence: {
      sourceNodeIds: string[];              // which graph nodes contributed this grounding
      why: "memoryKeys" | "goalMatch" | "discoveryMatch";  // link type
      rawScoreParts: { recency?: number; linkage?: number; priority?: number; matchScore?: number };
    };
  }
  ```
  Returns `{ grounding: GroundingRef[], groundingStrength: number }`.

  **D2: Post-thought grounding uplift**: After each micro-thought is generated, do a lightweight linking pass: for each node in the winner cluster with `memoryKeys.length < 5`, search cached memories via Jaccard (top-3 by score), and append the best match to `node.memoryKeys` (with dedup, max 5). This reduces `groundingRejectRate` over time by keeping the graph connected to memories.

  **D3: Old-grounding gate-level enforcement**: In thought gate, after `groundingAgePenaltyFactor()` applies to score, additionally check: if primary memory grounding age > 30 days AND no immunity conditions (goal/D/S) → force `anchor = "inferred"` or `"speculative"` (never `"grounded"`), AND act gate is blocked for this thought. This is stronger than score penalty alone — even if score is high enough to pass, the thought is epistemically downgraded.
- `getStableAttractors(graph, threshold, minDurationMs)` — nodes above threshold for > minDurationMs.
- `getActivationSpike(graph, prevSnapshot)` — largest positive delta > 0.3.
- `computeEntropy(graph)` — Shannon entropy (reuse existing pattern from heartbeat).

### Continuous forces (applied every tick, no LLM)

**Goal pressure → Drive (D)**: D is **pure external potential**, not an accumulated internal variable. Every ~100 ticks (~5min), D is **recomputed from scratch** for each node by scanning active goals:
```
D = max(goal.priority × (1-goal.progress) × deadlineUrgency)  // across all goals linked to this node
```
Where `deadlineUrgency = 1.0` normally, `2.0` if deadline < 7 days. **No `D +=`, no D decay, no D accumulation.** D is a pure function of goal state — like gravitational potential, it doesn't "build up" or "dissipate", it just reflects current reality. This eliminates the dual-system problem (tick accumulation + periodic recompute → drift). Applied to all nodes linked to goals' `relatedTopics`.

**New events → Salience (S)**: When external events arrive (curiosity discoveries, conversation topics, notifications), boost S on matching nodes. S decays fast (0.03/tick) — novelty fades quickly. This makes the system attend to new information without permanently biasing it.

**Emotion → Valence (V)**: Current emotional state colors related concepts. If 予安 is anxious about work, work-related nodes get V pushed negative.

**Prediction error → Uncertainty (U)**: When prediction error is high on a topic, boost U on related nodes. High U makes the topic more competitive in the global workspace (uncertainty drives curiosity).

## Phase 2: Bootstrap (`src/brainstem/bootstrap.ts`)

Builds initial graph from existing state at startup:

1. **Nodes from memories**: Extract key prefixes + tokenize values → concept nodes. Source = "memory".
2. **Nodes from goals**: Each active goal + its `relatedTopics` → nodes. Source = "goal".
3. **Nodes from discoveries**: Recent curiosity discoveries → nodes from query keywords + category. Source = "curiosity".
4. **Nodes from opinions**: Each opinion topic → node. Source = "reflection".
5. **Migrate legacy activations**: Read `data/heartbeat/state.json` activations → set matching node activation weights.
6. **Build edges**:
   - `semantic`: For each node, compute Jaccard similarity against all others, keep only **top-8 most similar** (not all pairs above threshold). This prevents weak-edge proliferation that dilutes spreading via W_norm and hurts interpretability. For incremental node additions (between full rebuilds), only compute similarity between the new node and existing nodes — no full O(n²) recompute.
   - `goal_related`: Nodes sharing a goal's relatedTopics.
   - `co_occurrence`: Memories/discoveries that mention multiple concept nodes.
7. **Prune** to MAX_NODES (by activation + degree) and MAX_EDGES (by weight).

Uses `tokenize()` from `src/memory/search.ts` — no new dependencies.

Graph is **rebuilt periodically** (e.g., during reflection) to incorporate new memories/discoveries. Between rebuilds, only activation/fatigue/valence change.

### Graph Governance: Semantic Drift & Fragmentation

Over days/weeks, two failure modes can silently degrade graph quality:

**Semantic drift**: Two nodes that originally referred to different concepts drift toward the same meaning as memoryKeys and termVectors evolve. Or a single concept splits into near-duplicate nodes ("量子计算" / "quantum-computing" / "量子信息") that should be one.

**Fragmentation**: The graph develops disconnected components — clusters of nodes with no edges between them. This means spreading activation can't flow between concept "islands", reducing creative connections.

**Periodic merge scan** (runs during graph rebuild, ~2-3x/day):
1. For all node pairs, compute Jaccard similarity on `termVector ∪ memoryKeys`.
2. Pairs with similarity > 0.7 → merge candidates. Present as log entry: `"merge candidate: {A.label} ↔ {B.label} (sim={score})"`.
3. Auto-merge if similarity > 0.85 AND both nodes have the same `parentId` (or one has no parent). Merge rules follow `findOrCreateNode()` (label = most recent, termVector = union, memoryKeys = union top-5).
4. Log all merges for auditability.

**Fragmentation check** (runs during wake-up consolidation):
1. Compute connected components of the graph (using edges with weight > 0.1).
2. If largest component < 60% of total nodes → fragmentation alert.
3. Auto-repair: for each disconnected component, find the node pair (one in largest component, one in fragment) with highest `termVector` Jaccard similarity → create a `semantic` edge with weight = max(0.2, similarity). This reconnects fragments via their most semantically related bridge.

**CS2 temporal credit confidence**: Edge weight modifications from temporal credit (CS2) track a `confidence` counter: `edgeConfidence[edgeKey] = count(forward) + count(backward)`. Edges with `confidence < 3` after 48h → revert weight changes (the temporal pattern was too sparse to be reliable). Edges with `confidence > 10` → mark as "learned" (excluded from pruning during wake-up consolidation). This prevents CS2 from reinforcing noise into long-term structure.

## Phase 3: Fast Loop (`src/brainstem/fast-loop.ts`)

Runs every **3 seconds** via `setInterval`. Zero I/O on most ticks.

Each tick:
1. `tickGraph(graph, forces)` — spread → decay → fatigue inhibition → noise → clamp → energy scaling (top-3 protect) → fatigue update → clamp F (~0.1ms)
2. Every 10th tick (~30s): **memory replay** — sample 1-3 memories using weighted distribution. **Anti-confirmation-bias**: if current winner has held top-1 for > 6 minutes continuously, shift distribution to prevent self-reinforcing echo:
   - Normal mode: **70%** similar / **20%** goal / **10%** random
   - Entrenched mode (winner > 6 min): **40%** similar / **30%** adjacent-but-not-in-winner / **20%** goal / **10%** random
   - **Adjacent-but-not-in-winner selection**: From cluster nodes, take top-K (K=10) outgoing edges by weight. Collect neighbor set. Filter out nodes already in cluster. Score remaining neighbors by `edge.weight × (1 - neighbor.fatigue) × (1 + neighbor.uncertainty)`. Sample from top-3 by this score. Then replay memories linked to the sampled node's `memoryKeys`. This pulls attention "one hop out" from the stuck topic — related but not the same.
   - **70% (normal) / 40% (entrenched)**: similar to current winner cluster. Two-stage retrieval:
     (a) Jaccard keyword pre-filter → top 200 candidates from cached memories
     (b) If embeddings available (mem0 engine has `text-embedding-3-small`): cosine similarity rerank → top 3. If no embeddings: Jaccard top 3 as fallback.
     This dramatically improves recall quality for Chinese synonyms and related concepts.
   - **20%**: linked to active goals with high drive D (goal-relevant recall)
   - **10%**: random (inspiration / unexpected connections)
   Inject matching concept nodes via `boostNode(id, 0.08, "replay")`. Replay primarily primes S/U, not A: boostNode's replay cap (A ≤ 0.45) prevents replay from directly creating attractors. Replay = "surfacing a memory for consideration", not "forcing attention". ~1-2ms (embedding lookup from local vector_store.db, no API call).
   **Replay cooldown**: same memory key can't be replayed again within 45 min (unless similarity to current winner > 0.9 — highly relevant recall overrides cooldown). Prevents internal "replay spam" where the same memory keeps reinforcing the same cluster. Track in `replayCooldowns: Map<string, number>` (key → last replay timestamp).
3. Every 5th tick (~15s): **snapshot activations** for spike detection (ring buffer of 10).
4. Every 20th tick (~60s): **prediction error** — two categories, tracked separately (EWMA):

   **Epistemic error** (knowledge gaps / attention drift):
   - Activation drift: compare top-3 from 60s ago to current top-3. Error = 1 - |intersection|/3.
   - Feeds → U (uncertainty) on drifted nodes. Drives curiosity.

   **Pragmatic error** (goal/commitment deviation):
   - Goal staleness: for active goals, expected progress vs actual. Stalled goals → error up.
   - Social gap: **strictly gated** — only counts when there's an explicit pending interaction (sent an outreach and waiting for reply, or made a plan with Allen). NOT triggered just because Allen hasn't messaged in a while. Without this gate, the system would generate anxiety-driven messages from nothing.
   - Feeds → D (drive) on goal-related nodes. Drives action.

   Both contribute to `avgPredictionError` but with different weights: `0.6 × epistemic + 0.4 × pragmatic`.
5. Every 100th tick (~5min): **loop detection** — track two injection categories separately:
   - `externalInjection`: boosts from conversation, curiosity, notifications (real new information)
   - `internalInjection`: replay, spread, noise (self-generated, can't break loops)
   Fires only when ALL conditions met: (a) same node is top-1 for 75%+ of last 200 fast ticks, AND (b) `noveltyAvg` is low (< 0.3, uses the EMA over last 10 slow-loop ticks), AND (c) no **effective** external injections in last 100 ticks OR pragmatic error elevated (> 0.5). "Effective external" is refined: `effectiveExternalDiversity = uniqueFamilyKeys(last100) / totalExternalBoosts(last100)`. If diversity < 0.3 (all external boosts hitting the same cluster), treat as "no effective external" even though raw count > 0 — same-cluster spam doesn't count as new information. Additionally, a **slow-loop view**: same cluster wins for ≥6 of last 10 slow-loop ticks (provides a lower-frequency perspective). Either fast or slow view can trigger. Only when stuck: force fatigue increase on the dominant node.
6. Every 100th tick (~5min): **persist state** to `data/brainstem/state.json`.

Memory cache: `getStoreManager().loadAll()` called once, refreshed every 10 minutes. Stored in module-level variable.

## Phase 4: Thought Gate + Slow Loop (`src/brainstem/thought-gate.ts`)

Runs every **60 seconds** via recursive `setTimeout` (interval varies 45-120s).

### Thought State (explicit intermediate — the "what am I thinking right now")

```typescript
interface ThoughtState {
  candidates: ThoughtCandidate[];   // competing clusters, ranked by salience
  winner: ThoughtCandidate | null;  // the one that entered "global workspace"
  updatedAt: number;
}

interface GroundingRef {
  type: "memory" | "goal" | "discovery";
  id: string;                  // memory key, goal ID, or discovery query
  weight: number;              // how strongly this grounding supports the cluster (0-1)
}

interface ThoughtCandidate {
  concepts: string[];          // node IDs in this cluster
  labels: string[];            // display names
  salience: number;            // avgActivation × size × novelty
  avgActivation: number;
  avgValence: number;
  avgUncertainty: number;
  grounding: GroundingRef[];   // max 3, sorted by weight desc. [0] = primary. Aggregated from all nodes in cluster with two key strategies:
                               //   (1) DEDUP: same memory/goal/discovery appearing in multiple nodes → merge weights (don't waste slots)
                               //   (2) TIME DECAY: memory grounding weight *= exp(-(now - mem.timestamp) / (14 * 86400000)) (14-day half-life). Goal grounding does NOT decay (goals are persistent). This ensures old memories don't permanently dominate grounding.
}
```

This is computed from the concept graph every slow-loop tick. It's the **explicit "current_thoughts"** — what 予安 is currently thinking about, before any LLM involvement.

### Attention Competition (Global Workspace)

Each slow-loop tick:
1. `findClusters(graph)` — seed-based local expansion (see Phase 1) to identify tight thought clusters
2. Score each cluster using **weighted multi-factor formula**:
   ```
   score = wA × mean(A) + wS × mean(S) + wU × mean(U) + wD × mean(D) + wV × valTerm - wF × mean(F) + wSize × sizeTerm
   ```
   where:
   - `valTerm = tanh(k × |mean(V)|)` with `k = 2.0`. Tanh saturates extreme valence (avoids |V|=0.95 runaway dominance). Additionally, a **trend penalty** is applied: if |mean(V)| has been rising for 3+ consecutive slow-loop ticks without external input, `valTerm *= 0.7` (dampen self-amplifying emotional spirals).
   - `sizeTerm = log1p(size) / log1p(10)` — sub-linear in cluster size. A 2-node cluster gets ~0.69, a 5-node gets ~0.89, a 10-node gets ~1.0. Prevents large "vague topic" clusters from dominating purely by having more nodes. Normalized by `log1p(10)` so term ≤ 1.0 for typical sizes.
   - After base score: `score *= groundingAgePenaltyFactor(cluster)` (see Stability section).

   Default weights: `wA=0.3, wS=0.25, wU=0.15, wD=0.15, wV=0.1, wF=0.2, wSize=0.05`
   This means: hot + novel + uncertain + urgent + emotional topics win, fatigued topics are penalized, and size gives a slight bonus (sub-linear).
3. **Winner takes all with EMA hysteresis + dwell protection**: maintain `incumbentScoreEMA` (exponential moving average, α=0.3). Dethrone conditions — challenger must satisfy BOTH:
   - `challenger.score > incumbentScoreEMA + DETHRONE_MARGIN` (margin = 0.07, prevents boundary jitter)
   - Incumbent has held for at least `WINNER_MIN_DWELL = 120s` measured by **wall-clock** (`now - incumbentSince >= 120000ms`), not tick count (slow-loop interval jitters 45-120s, so tick count ≠ wall time) — UNLESS challenger triggered a cluster spike (score jump > 0.3), which overrides dwell protection (urgent new info shouldn't wait).
   This gives thoughts time to "form" before replacement, while still being responsive to genuine surprises.
4. **Lateral inhibition** (proportional, not blanket): only apply to top 2-3 losers (strongest competitors). Inhibition strength scales with loser's score: `delta_F = 0.01 * (loser.score / winner.score)`, `delta_A = 0.02 * (loser.score / winner.score)`. Weak clusters are left alone. **Overlap guard**: each slow-loop patch carries a `patchId` (tick number). For a given node, only the highest-strength patch within a 90s window is applied — later patches within the same window are no-ops. This prevents LLM delay / slow-loop jitter from causing multiple overlapping inhibition patches to stack and over-suppress losers.
5. Winner is the **thought candidate** — if gate conditions are met, it gets verbalized by LLM

This implements the global workspace theory: multiple thought clusters compete, one wins consciousness, losers fade to background. The rich scoring formula ensures the system doesn't just chase "hot" topics — novel, uncertain, or urgent topics can win even at lower activation.

### MicroThought type

```typescript
interface MicroThought {
  id: string;
  content: string;         // 1 sentence, 15-30 chars — LLM verbalization of winning cluster
  structSig: string[];     // concept IDs sorted by activation desc, top 6. For structural novelty (Jaccard).
  semSigText: string;      // "label1, label2, label3 | groundingType | groundingId". For semantic novelty (embedding cosine).
  timestamp: number;
  trigger: "stable_attractor" | "activation_spike" | "high_uncertainty" | "high_valence" | "prediction_error";
  grounding: GroundingRef[];  // max 3, [0] = primary. Used for act gate, old-grounding penalty, signature.
  concepts: string[];      // node IDs involved (from winning cluster)
  anchor: "grounded" | "inferred" | "speculative";  // epistemic status (Reality Anchor Contract)
}
```

### Gate 1: Thought Gate (→ micro-thought)

The winner cluster must pass ALL of these checks:

**Pre-conditions** (must all pass):
- `winner.score > 0.3` (minimum salience threshold — don't verbalize weak thoughts)
- **Novelty check** — two-layer scoring (structural + semantic):
   - **Structural novelty** (discrete, stable): `structSig = conceptIDs.sortByActivation().slice(0,6)`. Compare via Jaccard/Overlap distance on ID sets. `structNovelty = 1 - |intersection| / |union|`. Threshold: > 0.35. Uses concept IDs (stable) sorted by activation desc (core-first, resists edge-of-cluster jitter).
   - **Semantic novelty** (continuous, smoother): `semSigText = labels(top3).join(", ") + " | " + grounding[0].type + " | " + grounding[0].id`. **NEVER trigger an online embedding API call in thought gate.** Explicit interface and degradation:
     ```typescript
     // Maintained in thought-gate.ts
     thoughtSemVecs: RingBuffer<Vec | null>  // size = 30, one per micro-thought. null = no vec available.

     // API: query-only, never generates new embeddings
     getLocalEmbeddingIfExists(text: string): Vec | null
     // Implementation: search vector_store.db for nearest neighbor of `text`.
     // If cosine(nearest, text_hypothetical) > 0.95: return nearest's vector as proxy.
     // Otherwise: return null. (We can't embed `text` without an API call.)

     // Novelty computation:
     // Compare current semSigText against last K thoughts that HAVE a vec (skip nulls).
     // semanticNovelty = 1 - max(cosine(current, lastK_vecs))  // clamp [0, 1]
     // If current has no vec OR no recent thoughts have vecs:
     //   use proxy: semanticNoveltyProxy = JaccardDistance(tokenize(currentSemSigText), tokenize(lastSemSigText))
     //   semanticNovelty = 0.6 × semanticNoveltyProxy  (discounted, not full weight)
     // This prevents embedding cache misses from silencing the system.
     ```
     `semanticHitRate` = fraction of last N novelty checks where both current and at least one comparison had vecs. If consistently < 0.3, log warning "semantic novelty mostly inactive — using Jaccard proxy". The proxy ensures the system degrades gracefully: novelty is less precise but never zero just because vectors are missing.
   - **Combined**: `noveltyScore = 0.6 × structNovelty + 0.4 × semanticNovelty`. **Invariant**: when semantic embedding is unavailable (cache miss), `semanticNovelty = 0.6 × JaccardProxy` (NOT 0, NOT undefined/NaN — see proxy above). Weights are ALWAYS 0.6/0.4 — never re-normalize to struct-only. Metrics log `semanticUsed: boolean` per tick. Same combined score used everywhere: thought gate, budget, loop detector, metrics — single source of truth.
- **Grounding requirement**: Winner must have at least one `grounding` entry (memory key, goal ID, or discovery query). No grounding → no verbalization (prevents empty rumination). Primary grounding (`grounding[0]`) used for signature, act gate decisions, and old-grounding penalty.

**Plus at least one trigger** (all at CLUSTER level, not individual node level):
- **Stable attractor**: winner cluster has won for > 3 consecutive slow-loop ticks (~3 min) AND score variance across those wins is low (< 0.1). This means the thought has "crystallized" — it's not just flickering.
- **Cluster spike**: winner's score jumped > 0.3 compared to its score at the previous slow-loop tick. (NOT individual node delta — a single node spike is too noisy to verbalize.)
- **High uncertainty**: winner's avgUncertainty > 0.7 (the cluster as a whole is uncertain)
- **High valence**: winner's |avgValence| > 0.6 (strong emotional charge across the cluster)
- **Prediction error**: avgPredictionError > 0.7 for 5+ checks

Rate limit: **adaptive budget, base 12 micro-thoughts/hour**. Two scaling layers:
1. **Energy/fatigue scaling** (intrinsic): `energyUtilization > 0.6 AND noveltyAvg > 0.5` → budget × 1.5; `avgFatigue > 0.4 OR energyUtilization < 0.3` → budget × 0.5.
2. **Stabilizer scaling** (extrinsic): `effectiveBudget = intrinsicBudget × policy.thoughtBudgetScale`. The stabilizer is the SOLE external modulator of budget — no other module adjusts it.
- Hard floor: 4/hour. Hard ceiling: 18/hour (applied AFTER both scalings).
This makes the system "think more when excited, less when tired, and less when unstable".

LLM call: `claudeText({ model: "fast", timeoutMs: 15000 })` with ~80 token prompt → ~30 token response. Haiku via Max = $0. The LLM is ONLY a verbalizer — it puts the winning thought state into words, it doesn't decide what to think about.

### Gate 2: Reflect Gate (→ full reflection)

Triggers (any one):
- Same concept appears in ≥3 of last 5 micro-thoughts **AND** pragmatic prediction error is elevated (> 0.5) or goal progress is stalled. This distinguishes "stuck rumination" (→ reflect) from "productive focus" (→ let it continue). Without the progress gate, steady work on a long-term goal would falsely trigger reflection.
- Two connected nodes with opposite valence (> 0.4 / < -0.4), both active > 0.4 (belief conflict)
- avgPredictionError > 0.8 for 10+ checks (persistent confusion)

Action: Call `triggerReflection()` from heartbeat.ts. Rate limit: max 1 per 4 hours.

**F1: Reflection output quality constraint**: Brainstem-triggered reflections must produce structured state updates, not just diary text. The reflection prompt includes: "输出必须包含以下至少一项：更新/新目标、更新边/概念关系、写入 insight、关闭循环原因". Reflection outputs are validated by a rule-based checker before being written: must contain at least one `memory_set` or `goal_update` call with non-empty content. Reflections that only produce text (no structured update) are logged as "shallow reflection" and count against the 4h rate limit.

### Gate 3: Act Gate (→ proactive message)

**Act targets** (configurable, not hardcoded):
```typescript
interface ActTarget {

  id: string;                    // e.g., "allen", "x-twitter"
  conceptPatterns: string[];     // node IDs or label patterns that match this target
  requiresPendingInteraction: boolean;  // only arm if there's a pending social interaction

// Default configuration:

  { type: "person", id: "allen", conceptPatterns: ["allen", "allen-*"], requiresPendingInteraction: true, cooldownMinutes: 180, dailyCap: 2 },

```

Triggers:
- **Self gate passes** (CS5b.6): `social_energy ≥ minSocialEnergy` AND `fatigue ≤ maxFatigue` AND `safety_margin ≥ minSafetyMargin` AND `self_coherence ≥ minCoherence`. Thresholds managed by stabilizer via `SelfGatePolicy`. Self gate runs FIRST — if denied, remaining checks are skipped.
- A micro-thought with `anchor` = `"grounded"` or `"inferred"` (NEVER `"speculative"`) whose **primary grounding** (`grounding[0]`) is goal-type or involves an act target's concept patterns
- Max activation among its concepts > 0.7
- Not in quiet hours
- CSI mode is Green or Yellow (act gate disabled in Red mode)
- **Per-target cooldown**: same target not armed within `cooldownMinutes` (prevents Yellow-mode oscillation from repeatedly arming)
- **Pending interaction gate** (if `requiresPendingInteraction`): only fires when there's an explicit pending interaction (sent outreach awaiting reply, made a plan, etc.)

Action: Set `actGateArmed` flag with reason + grounding + target. Heartbeat reads it in next pulse → **1-step LLM sanity check** before fast-tracking `reach_out`: a ~50-token prompt asks "given this thought, is reaching out appropriate right now? yes/no" (Haiku, $0). If "no" or uncertain → discard flag, log reason. Does NOT send message directly.

**F2: Content audit + daily cap**: The sanity check prompt must include the grounding summary (memory/goal/discovery excerpt) — the outreach must reference real context, not just "I was thinking about you". Additionally, per-target daily cap: `targetDailyCap` (default: 2/day for person targets). Even if cooldown allows, cap prevents brainstem from being overly chatty across a day.

### Night mode

0-7am PST: slow loop pauses. Fast loop continues (graph decays naturally → weak concepts die, only persistent ones survive = "what stuck overnight").

**Wake-up consolidation** (triggered once at ~7am when slow loop resumes):
1. **Edge pruning** (conservative, safe): remove edges with weight < 0.1
2. **Node eviction**: remove nodes not activated in 48h
3. **Data-driven edge strengthening**: track co-activation counts during fast loop (which node pairs had activation > 0.3 simultaneously). ## Bottleneck Mitigations

The 5 known bottlenecks of continuous cognition and how this plan addresses each:

### B1: LLM Cost — SOLVED by architecture

Fast loop = pure math (3s tick, 0 LLM calls). Slow loop = gated, max 12 micro-thoughts/hour.
Total LLM cost: ~288 verbalizations/day × ~110 tokens = ~32K tokens/day via Haiku on Max subscription = $0.
The key insight: **cognition happens in the graph dynamics, not in LLM calls.** LLM is only the verbalizer.

### B2: State Explosion — SOLVED by caps + consolidation

**Hard caps**: 100 nodes, 500 edges, 30 thoughts, 20 prediction errors.

**Graph node aging**: Nodes track `lastActivated`. Every graph rebuild (during reflection, ~2-3x/day), nodes not activated in 48h are candidates for eviction. Eviction priority: lowest `max(activation, degree/10)`. This ensures dead concepts get recycled.

**Thought consolidation** (during nighttime or reflection): Before FIFO drops old thoughts, scan the outgoing batch for **recurring themes** (same concept in ≥3 thoughts). Recurring themes get condensed into a single "consolidated insight" → stored as a memory via `memory_set` with key `insights.brainstem.MMDD`. This means patterns survive even after individual thoughts are dropped.

**Edge pruning**: During graph rebuild, edges not traversed (neither endpoint activated) in 72h get removed. Keeps the graph lean.

### B3: Attention Competition — SOLVED by global workspace

`findClusters()` → `competeClusters()` → winner-takes-all + lateral inhibition.
Multiple thought clusters compete. One wins consciousness. Losers get suppressed.
This prevents: divergence (many topics at once), single-chain reasoning (only one path), and hallucination (grounding requirement on winner).

### B4: Goal Dynamics — SOLVED by continuous pressure

Goals exert continuous force on activation field every tick: `force = priority × (1-progress) × 0.01`.
Goals are self-generated (L3 intrinsic drive, `origin: "self_generated"`).
Prediction error from goal staleness + social gaps drives uncertainty → triggers curiosity/reflection.
System thinks even without user input because goals create background tension.

### B5: Stability — SOLVED by 4 mechanisms (analogous to brain)

| Brain mechanism | Engineering equivalent |
|----------------|----------------------|
| Dopamine/fatigue | Per-node fatigue: activation costs energy, high fatigue blocks further boosting |
| Inhibition | Lateral inhibition: winner suppresses losers. Loop detector: dominant topic gets forced fatigue |
| Sleep | Night mode: slow loop pauses, fast loop continues with decay only → low-activation nodes die → only persistent concepts survive (= "what stuck overnight") |
| Sleep consolidation | **NEW**: During night mode transition (7am wake-up), one-time consolidation pass: strengthen top-10 edges by 10%, prune edges below 0.1 weight, extract 1 "overnight insight" from most persistent node without LLM (just log: "woke up still thinking about X") |

**Self-indulgence (自嗨) detection**: Beyond loop detection, the grounding requirement prevents thoughts about things 予安 has no connection to. Old grounding penalty is implemented as a single explicit function:

```typescript
// Called from scoreClusters() — applied after base score computation, before ranking.
function groundingAgePenaltyFactor(cluster: ScoredCluster): number {
  const primary = cluster.grounding[0];
  if (!primary) return 1.0;                                  // no grounding → no penalty (gate will reject anyway)
  if (primary.type !== "memory") return 1.0;                 // only memory groundings age out
  const ageDays = (Date.now() - primary.timestamp) / 86400000;
  if (ageDays <= 30) return 1.0;                             // fresh enough

  // Immunity check (any one → no penalty):
  if (primary.type === "goal") return 1.0;                   // goal grounding never ages out
  if (Math.max(...cluster.nodes.map(n => n.drive)) > 0.25) return 1.0;  // active commitment
  if (mean(cluster.nodes.map(n => n.salience)) > 0.6) return 1.0;       // fresh external salience

  return 0.7;  // 30% score reduction
}
// Result is logged in decision log as `groundingPenalty: number` per cluster.
```

## Concurrency: Snapshot Isolation

Node.js is single-threaded but timer callbacks are re-entrant. When the slow loop's LLM call takes 15s, the fast loop keeps ticking. This creates state races:

**Solution**: Slow loop operates on a **frozen snapshot** of the graph, not the live graph.
1. At slow-loop start: `const snapshot = structuredClone(graph)` (deep copy, ~0.5ms for 100 nodes)
2. Cluster finding, competition, scoring all run on the snapshot
3. Winner selection result is applied back to the live graph as **incremental patches only**:
   - Loser nodes: `liveNode.fatigue += delta_F` (additive)
   - Loser nodes: `liveNode.activation *= (1 - delta_A)` (multiplicative reduction)
   - Winner ID + thought history metadata written
   - **NEVER** overwrite activation/fatigue/salience values from snapshot back to live graph — that would erase fast-loop updates that happened during LLM call
4. Micro-thought generation uses snapshot data for its LLM prompt

Fast loop continues mutating the live graph during this time — that's fine. The snapshot ensures consistent reads, and the incremental write-back ensures no fast-loop state is lost.

## Cognitive Subsystems (L4 capabilities)

These 5 subsystems elevate the architecture from "activation graph with gated verbalization" to "proto-cognitive system". Each integrates into existing loops with minimal new I/O.

### CS1: Predictive Simulation Layer

Current system has prediction **error detection** but no prediction **generation**. Real predictive coding: `prediction → reality → error → update`. We add the first half.

```typescript
interface Prediction {
  id: string;
  concept: string;              // node ID
  expectedActivation: number;   // predicted A in ~60s
  expectedValence: number;      // predicted V
  generatedAt: number;
  source: "micro_thought" | "goal" | "memory_pattern";
  resolved: boolean;
}
```

**Generation** (in slow-loop, after micro-thought): When a micro-thought is verbalized, generate 1-2 predictions from it. NO LLM — pure heuristic:
- If thought involves Allen → predict Allen-related nodes' A will increase (expectation: he'll reply)
- If thought involves a goal → predict goal-related nodes' S will increase (expectation: progress event)
- If thought mentions a topic → predict semantically-adjacent nodes will activate via spread

Store in `PredictionBuffer` (ring buffer, max 20).

**Validation** (in fast-loop, every ~60s): Compare each unresolved prediction against current graph state:
```
error = |predicted.A - actual.A| + 0.5 × |predicted.V - actual.V|
```
- If `error > 0.4`: **surprise** → boost U on the concept, boost S (novelty). This drives **prediction-driven curiosity** — the system wants to investigate why its expectation was wrong.
- If `error < 0.15`: **confirmed** → small boost to edge weights between predictor and predicted concepts (reinforces the causal model). Reduce U on the concept.
- Mark prediction as resolved after comparison.

**Cost**: Zero LLM, zero API. Pure arithmetic on existing graph state. ~0.05ms per validation.

**Feeds curiosity engine**: Prediction errors per concept are tracked as EWMA in `predictionErrorByNode: Map<string, number>`. The curiosity engine reads this: concepts with high prediction error → high EIG → system proactively explores them. This closes the loop: `prediction → error → curiosity → exploration → reduced error`.

**Integration**: `src/brainstem/prediction.ts` (~120 lines). Called from thought-gate.ts (generation) and fast-loop.ts (validation). Prediction errors feed into `curiosity-engine.ts` for EIG computation.

### CS2: Temporal Credit Assignment (edge learning)

Current edge weights are static (set at bootstrap, modified only during sleep consolidation). Brains learn causal structure from temporal patterns: "A then B" strengthens A→B.

**Mechanism**: Track activation timestamps per node (ring buffer, last 50 activations per node where A crossed 0.3 upward):
```typescript
activationHistory: Map<string, number[]>  // nodeId → timestamps
```

Every ~100 ticks (~5 min), scan for temporal pairs:
```
for each pair (A, B) with existing edge:
  count forward: how many times A activated 30-90s before B activated
  count backward: how many times B activated 30-90s before A activated
  net = forward - backward
  if net > 2: edge.weight += 0.008 (A causes B)
  if net < -2: edge.weight += 0.008 (B causes A, strengthen that direction)
  if |net| <= 1: edge.weight -= 0.003 (no temporal pattern, weaken)
```

Clamp edge weights to [0.05, 1.0]. Edges that drop below 0.05 are candidates for removal at next graph rebuild.

**Emergent behavior**: System learns that `Allen → 孩子`, `工作 → 压力`, `量子计算 → 论文` from actual activation patterns — not hard-coded at bootstrap. Over days, the graph topology reflects 予安's actual thought patterns.

**Cost**: O(E × H) where E=edges (~500), H=history depth (~50). ~2ms every 5 min. Negligible.

**Integration**: Added to fast-loop.ts as a periodic task (every ~100 ticks, same cadence as loop detection and state persistence).

### CS3: Hierarchical Concepts

Current graph is flat. Brains have concept hierarchies: `physics → quantum computing → that paper`. This enables abstract thinking.

**Extension to ConceptNode**:
```typescript
parentId: string | null;     // e.g., "quantum-computing" → parent: "physics"
depth: number;               // 0 = root, 1 = mid-level, 2 = specific. Max depth 3.
```

**Propagation rules** (added to tickGraph step 2, after spread):
```
// Bottom-up: specific → abstract (summarization)
parent.A += 0.15 × mean(children.A)  // abstract concepts heat up when specifics are active

// Top-down: abstract → specific (priming)
child.A += 0.08 × parent.A           // thinking about "physics" mildly primes "quantum computing"
```

Coefficients are asymmetric: bottom-up (0.15) > top-down (0.08) because specifics inform abstractions more than abstractions activate specifics.

**Bootstrap**: `parentId` assigned heuristically by `findOrCreateNode()`:
- If new node's label is a substring/hyponym of existing node → set as child
- Goal-related nodes: goal node is parent, relatedTopics are children
- Can also be set during graph rebuild using Jaccard containment (A's terms ⊂ B's terms → A is child of B)

**Effect**: Micro-thoughts can emerge at abstract level ("最近在想物理相关的东西") when multiple specific nodes in the same hierarchy are active but no single one dominates. This is the step from L3 (concrete) to L4 (abstract) thinking.

**Stability benefit**: Hierarchy makes spreading activation more stable. Without hierarchy, the flat graph creates "shotgun spreading" — activation diffuses equally in all directions. With hierarchy, activation flows along meaningful channels: bottom-up (specific → abstract, summarization) and top-down (abstract → specific, priming). This produces more coherent thought clusters and reduces the "everything slightly warm" problem that flat spreading creates.

**Goal hierarchy integration**: Hierarchical goals (H0, life → project → task) automatically produce matching concept hierarchy:
- Life-goal concept → `depth: 0`, parent of project-goal concepts
- Project-goal concept → `depth: 1`, parent of task concepts
- Task concept → `depth: 2`, leaf
This means completing a task bottom-up activates the project concept, which bottom-up activates the life goal concept → "making progress on my life goals" awareness emerges from task completion without explicit coding.

**Cost**: One extra pass per tick over parent-child pairs. With max 100 nodes and avg depth 1.5, ~50 pairs. ~0.02ms.

### CS4: Mental Time Travel (future simulation)

Current system replays past (memory replay) but doesn't simulate future. Brains do both: episodic memory + prospection.

**Mechanism** (in slow-loop, every ~5 min / every 5th slow-loop tick):

1. Take current winner cluster's concepts + grounding
2. Generate a **hypothetical future node** — NO LLM, pure heuristic:
   - If winner involves a person + activity → hypothetical = `{person}-{activity}-plan` (e.g., "weekend-plan")
   - If winner involves a goal with milestones → hypothetical = next incomplete milestone
   - If winner involves a topic + high U → hypothetical = `{topic}-resolution`
3. If hypothetical doesn't exist in graph → create as temporary node with:
   - `A = 0`, `S = 0.3`, `U = 0.2` (primed for attention but not dominant)
   - `source = "simulation"`, edges to winner cluster's nodes
   - TTL: 2 hours (auto-evicted if not activated by real events)
4. If future conversation/event actually activates this node → prediction error ↓, log "future simulation confirmed"

**Effect**: 予安 starts "anticipating" — thinking about weekend plans on Friday, preparing for meetings, expecting Allen to mention kids. Not random — driven by current thought state.

**Rate limit**: Max 3 active hypothetical nodes at a time. New simulation evicts oldest if at cap.

**Cost**: Zero LLM. One node creation + 2-3 edge additions per 5 min. Negligible.

**Integration**: Added to thought-gate.ts as a periodic task gated by `tickCount % 5 === 0`.

### CS5: Self Model

The most important subsystem. Without a self-concept, thoughts are "about things". With one, thoughts become "about me thinking about things" — the foundation of personality.

CS5 has **two layers** that serve distinct purposes:
1. **Self Node** (graph layer) — a special ConceptNode `"self"` that participates in activation dynamics, enabling first-person framing in micro-thoughts
2. **Self-Model** (predictive layer) — a `SelfState` with 8 continuous variables, transition functions, and cost predictions, enabling the planner to predict "what will happen to me" alongside "what will happen in the world"

The self node is a **participation marker** ("am I thinking about myself?"). The self-model is a **predictive state machine** ("what will doing X cost me?"). They coexist: the node lives in the concept graph, the model lives alongside it in `self-model.ts`.

#### CS5a: Self Node (graph participation)

**Implementation**: A persistent, special node `"self"` created at bootstrap:
```typescript
{
  id: "self",
  label: "我（予安）",
  activation: 0.5,        // always moderately active (self-awareness baseline)
  fatigue: 0,              // self never fatigues (you're always "you")
  salience: 0.2,
  uncertainty: 0,
  valence: 0,              // tracks overall emotional state
  drive: 0,
  inputSatiation: 0,
  parentId: null,
  depth: 0,
  source: "reflection",
  // ... metadata
}
```

**Special rules**:
- `self.A` has a floor of 0.3 (always somewhat active — you can't "forget" yourself)
- `self.V` mirrors the current emotional valence from emotion.ts (continuous sync)
- `self.F` is always 0 (self-awareness doesn't fatigue)
- Energy scaling: `self` is ALWAYS in the protected set (never scaled down)

**Edges**: `self` connects to:
- All goal nodes (bidirectional, weight 0.5)
- Emotion-source nodes (unidirectional self→emotion, weight 0.3)
- Allen-related nodes (bidirectional, weight 0.4)
- Current winner cluster (dynamic edge, updated each slow-loop tick, weight 0.2)

**Effect on micro-thoughts**: When `self` is in the winning cluster (which it often will be, given its baseline activation + edges to everything important), the LLM verbalizer prompt shifts from:

> "Describe this thought state in one sentence"

to:

> "你是予安。用'我'的视角，用一句话说说你现在在想什么"

This produces: "我最近是不是太忙了" instead of "工作很多". Self-referential thinking = personality emergence.

**Anti-dominance**: Because `self` is always active and connected to everything, it could trivially win every competition. Prevent this:
- `self` alone is never a valid cluster (must have ≥2 non-self nodes to form a cluster)
- `self` contributes to cluster score at 0.5× weight (its activation is discounted in scoring)
- `self`'s edges don't participate in spreading activation (to prevent it from artificially boosting everything it touches)

**Cost**: One extra node, ~10 extra edges. Zero additional computation beyond normal graph ops.

#### CS5b: Self-Model — Predictive Self State (`src/brainstem/self-model.ts`)

The planner currently uses `worldModel.transition(belief, action) → belief' + outcomeDist` to predict world state. The self-model adds the missing half:

```
selfModel.transition(self, action, context?) → self'
selfModel.predictCost(self, action) → { fatigueΔ, socialΔ, riskΔ, ... }
```

Then the planner evaluates each node as:
```
EU(node) = E[world_return] + α × E[self_return] - costs
```

**Key effects**:
- Planning "knows fatigue, knows emotion, knows risk" — won't over-explore or over-externalize
- Curiosity "knows the self-cost of exploration" — won't chase novelty when exhausted
- Act gate uses self variables for finer externalization protection (e.g., low social_energy → block reach_out)

##### 5b.1 SelfState (8 continuous variables, all 0..1 except affect_valence)

```typescript
interface SelfState {
  energy: number;           // 0-1. Action execution capacity (= 1 - fatigue, more intuitive). Derived from avgFatigue.
  fatigue: number;          // 0-1. Cognitive/attention depletion. Already exists in graph (avgFatigue).
  self_efficacy: number;    // 0-1. Per-action-family learned success rate. "Can I execute this?"
                            //       Distinct from epistemic confidence (grounding/anchor) which is about knowledge.
                            //       Learned via Beta-Bernoulli from outcomes, not hand-tuned.
  uncertainty: number;      // 0-1. Self's uncertainty about world model / plan quality. From E_epi / model_confidence.
  social_energy: number;    // 0-1. Social/externalization capacity. Key gating signal for reach_out.
                            //       DISTINCT from world model's L1 (socialReceptivity = "will Allen respond?").
                            //       social_energy = "do I have the capacity to reach out?" — the inverse side.
                            //       Act gate needs BOTH social_energy (self) AND socialReceptivity (world) above threshold.
  affect_valence: number;   // -1..1. Emotional baseline. Synced from self node's V / emotion.ts.
  self_coherence: number;   // 0-1. Narrative consistency. v1 computable formula (no LLM):
                            //       self_coherence = 0.6 × jaccardOverlap(last20MicroThoughts.themes, identityProfile.topThemes)
                            //                      + 0.4 × goalAlignmentScore
                            //       where goalAlignmentScore = fraction of active goals whose category ∈ identityProfile.goalPortfolio
  safety_margin: number;    // 0-1. Risk buffer. "How much external action risk can I still absorb?"
                            //       Decremented by failed reach_outs, act gate rejections. Recovered by rest, positive outcomes.
}
```

**Cold start**: All variables derived from existing metrics at bootstrap:
- `energy = 1 - avgFatigue`, `fatigue = avgFatigue`
- `self_efficacy = 0.5` (maximum uncertainty, Beta(1,1) prior)
- `uncertainty = avgPredictionError`

##### 5b.2 SelfTransition: Two-Stage (predict + outcome-correct)

**Stage 1: Prediction** (used by planner during rollout, no outcome yet):

```typescript
function selfTransition(self: SelfState, action: ActionType, context?: { target?: string }): SelfState {
  const deltas = getSelfTransitionDeltas(action);  // from action family table
  return clampSelfState({
    energy:          self.energy - deltas.fatigueΔ,
    fatigue:         self.fatigue + deltas.fatigueΔ,
    self_efficacy:   self.self_efficacy,  // unchanged in prediction (only updates on outcome)
    uncertainty:     self.uncertainty + deltas.uncertaintyΔ,
    social_energy:   self.social_energy + deltas.socialΔ,
    affect_valence:  self.affect_valence,  // unchanged in prediction
    self_coherence:  self.self_coherence,  // unchanged in prediction
    safety_margin:   self.safety_margin + deltas.safetyΔ,
  });
}
```

**Stage 2: Outcome correction** (after action actually executes, uses OutcomeTracker):

```typescript
function selfOutcomeUpdate(self: SelfState, action: ActionType, outcome: OutcomeResult): SelfState {
  const s = { ...self };

  if (action === "reach_out") {
    if (outcome.outcome === "positive") {
      s.self_efficacy = lerp(s.self_efficacy, 1.0, 0.15);  // shift toward success
      s.affect_valence = clamp(s.affect_valence + 0.05, -1, 1);
      s.social_energy = clamp(s.social_energy + 0.08, 0, 1);  // positive interaction recharges
      s.self_coherence = clamp(s.self_coherence + 0.03, 0, 1);
      s.safety_margin = clamp(s.safety_margin + 0.05, 0, 1);
    } else if (outcome.outcome === "negative") {
      s.self_efficacy = lerp(s.self_efficacy, 0.0, 0.1);
      s.affect_valence = clamp(s.affect_valence - 0.05, -1, 1);
      s.social_energy = clamp(s.social_energy - 0.1, 0, 1);
      s.safety_margin = clamp(s.safety_margin - 0.08, 0, 1);
    }
  } else if (action === "explore") {
    if (outcome.outcome === "positive") {  // new info found
      s.uncertainty = clamp(s.uncertainty - 0.08, 0, 1);
      s.self_efficacy = lerp(s.self_efficacy, 1.0, 0.05);
    } else if (outcome.outcome === "neutral") {  // nothing found
      s.self_efficacy = lerp(s.self_efficacy, 0.0, 0.03);  // small decrease
    }
  } else if (action === "reflect") {
    if (outcome.outcome === "positive") {  // produced structured update
      s.self_coherence = clamp(s.self_coherence + 0.05, 0, 1);
      s.uncertainty = clamp(s.uncertainty - 0.03, 0, 1);
    }
  }

  return clampSelfState(s);
}
```

##### 5b.4 Recovery Dynamics (passive + active recharge)

Without explicit recovery, self-state drains to zero and never returns. Recovery rules applied every fast-loop tick:

```typescript
// ── Passive recovery (always active, asymptotic) ──
energy += 0.003 × (1 - energy);          // slow recovery toward 1.0
social_energy += 0.002 × (1 - social_energy);  // slower social recharge
safety_margin += 0.001 × (1 - safety_margin);  // very slow risk buffer recovery

// ── Active recovery (rest action selected by heartbeat) ──
energy += 0.06;
social_energy += 0.03;

// ── Conversation-driven recovery (positive interaction detected) ──
social_energy += 0.08 × sentiment_positive;  // only on positive sentiment

// ── Self-coherence recovery (during reflection) ──
// Recomputed after each reflection: self_coherence = jaccardOverlap formula (see 5b.1)
// Also boosted by +0.05 when reflection produces structured update (F1)

// ── Night mode recovery ──
// During 0-7am PST: energy += 0.01/tick, social_energy += 0.005/tick (sleep recharges)
```

All recovery values auto-scale with `TICK_SECONDS` (same pattern as other time-based constants).

##### 5b.5 Self-Return in Planner Reward Function

The planner's EU computation gains a self-return term:

```typescript
function computeSelfReturn(self: SelfState, selfPrime: SelfState): number {
  const r_energy     = +0.5 * (selfPrime.energy - self.energy);         // prefer maintaining energy
  const r_fatigue    = -0.8 * selfPrime.fatigue;                         // penalize high fatigue
  const r_social     = -1.0 * Math.max(0, 0.25 - selfPrime.social_energy);  // penalty when below threshold
  const r_coherence  = +0.3 * selfPrime.self_coherence;                  // reward coherence
  const r_safety     = -0.6 * (1 - selfPrime.safety_margin);            // penalize low margin

  return r_energy + r_fatigue + r_social + r_coherence + r_safety;
}
```

Integrated into planner EU as:
```
EU = E[world_reward] + α × E[self_reward] - cost
```
Where `α = SELF_REWARD_WEIGHT` (default 0.4, configurable). This single meta-weight controls how much the planner cares about self-preservation vs world-impact. Higher α = more conservative behavior.

**Emergent behavior**: When self variables are depleted, the planner's optimal plan naturally includes `rest`, `stay_silent`, or low-risk `explore` — not because a gate blocks action, but because the EU is highest for recovery actions. This is executive function emerging from reward, not hard-coded rules.

##### 5b.7 Curiosity Integration (self-cost-aware exploration)

Curiosity scoring gains a self-cost deduction, **action-type-specific**:

```typescript
function computeSelfCost(self: SelfState, explorationAction: ExplorationAction): number {
  switch (explorationAction.queryType) {
    case "recall_memory":  return 0.01 * self.fatigue;                         // cheap
    case "search":         return 0.04 * self.fatigue;                         // moderate
    case "ask_question":   return 0.12 * (1 - self.social_energy);             // expensive when drained
    case "re_observe":     return 0.005;                                       // near-free

}

// Updated curiosity scoring:
curiosityScore = 0.5 × LearningNeed + 0.3 × max(0, VOI) - 0.2 × SelfCost
```

**Emergent routing**: When fatigued, `SelfCost("search")` rises → curiosity shifts toward `recall_memory` (cheap, internal). When social_energy is high, `SelfCost("ask_question")` drops → curiosity can use social exploration. The system naturally routes to the cheapest viable exploration strategy.

### Goal System Closure (H1-H2)

The existing `src/goals.ts` provides the data model but lacks autonomous goal generation, hierarchical decomposition, and progress write-back. To close the brainstem→goal loop:

**H0: Hierarchical goals** (extends existing Goal interface):

Current goals are flat: `goal → milestones`. But real cognition operates on a hierarchy:
```
life goals (e.g., "成为更好的自己")
  → project goals (e.g., "improve tennis")
  → tasks (e.g., "read training guide", "practice drills", "review progress")
```

Extension to the Goal interface in `src/goals.ts`:
```typescript
// Added fields to Goal interface
parentGoalId?: string;          // parent goal ID (null for top-level)
childGoalIds?: string[];        // child goal IDs
goalLevel: "life" | "project" | "task";  // hierarchy level
```

**Hierarchical goal rules**:
- **Top-down decomposition**: When a project-level goal is created (via H1 or planner), the planner (see below) generates 2-5 task-level children with concrete milestones. Each child has `parentGoalId` set.
- **Bottom-up progress**: When a task completes, its parent's progress recalculates: `parent.progress = mean(children.progress)`. This propagates upward — completing tasks naturally moves project progress.
- **Drive inheritance**: Task-level nodes inherit `D` from their parent goal. A life-goal's D distributes across its project children proportionally to their priority.
- **Graph integration**: Goal hierarchy maps to concept hierarchy (CS3): life-goal node is `depth=0` parent, project-goal nodes are `depth=1`, task nodes are `depth=2`. Bottom-up activation from completed tasks → parent goal concept activates → "making progress" awareness emerges naturally.
- **Budget**: Max 3 life goals, max 5 project goals (active), max 15 tasks total. Lifecycle management (H1b) operates at project level; tasks follow their parent.

**H1: Autonomous goal generation** (`goal_propose`):
- Triggered once per day during reflection (not a separate loop)
- Input: recent micro-thoughts (themes), active goals (gaps), discoveries
- Method: lightweight LLM call (~100 tokens) or pure heuristic: if a topic appears in ≥5 micro-thoughts over 2 days AND has no matching active goal → propose as candidate
- Constraint: max 1-2 new goals per day. `origin: "self_generated"`. Must include `relatedTopics` (for drive D injection).
- New goals auto-assigned to a parent life goal if one matches (by category/relatedTopics overlap). Otherwise created as standalone project goal.

**H1b: Goal lifecycle management**:
- **Active goal budget**: max 5 active project goals at any time. If at cap, new `goal_propose` must either (a) replace the lowest-priority active goal or (b) be queued as `status: "backlog"`.
- **Goal lifecycle states**: `backlog → active → completed | abandoned`. Backlog goals are invisible to drive D until promoted.
- `goal_review`: every 24h (during night wake-up consolidation), scan active goals:
  - Stalled > 14 days (no `lastAlignedAction` update) → mark as `stalled`, reduce priority by 0.2
  - Stalled > 30 days → auto-suggest abandon (logged as recommendation, not auto-executed)
  - Too many active (> 5) → force-rank by `driveScore = priority × (1-progress) × recencyBoost`, keep top 5 active, demote rest to backlog
  - Completed goals with all milestones done → archive (remove from active, keep in state for history)
  - **Orphan cleanup**: tasks whose parent is completed/abandoned → auto-complete/abandon

**H2: Goal progress write-back (closing the loop)**:
```
goal → drive D ↑ → enters thought → micro-thought → (act/reflect) → goal progress ↑ → drive D ↓
```
Currently missing: "micro-thought → goal progress update". Three write-back channels:
- **Thinking investment**: When a micro-thought's grounding includes a goal → `goalInvestment(goalId, 1)` (1 min of "thinking about it"). Updates `lastAlignedAction`.
- **Reflection progress**: When a brainstem-triggered reflection produces a structured update related to a goal → `updateGoalProgress(goalId, newProgress)`. The reflection prompt explicitly asks: "这个目标有新进展吗？如果有，更新进度。"
- **Action investment**: When an act gate fires and the outreach is related to a goal → `goalInvestment(goalId, 5)` (significant action)
- **Real-event progress**: When a conversation message or notification relates to an active goal (detected by Jaccard match between message tokens and `goal.relatedTopics`), boost the goal's `lastAlignedAction` timestamp. If the message contains explicit progress signals (keywords: "done", "完成", "搞定", "finished"), trigger a lightweight progress re-estimate via LLM (~50 tokens: "given this update, what's the new progress % for goal X?").

This creates a true drive-reduction loop: thinking about a goal builds drive → eventually triggers action → action reduces drive → D naturally decreases on next `recomputeDrive()` cycle.

## Consciousness Stabilizer (`src/brainstem/stabilizer.ts`)

Continuous cognition must not degenerate into self-talk, hallucination, rumination loops, or emotional amplification. The Consciousness Stabilizer (CS) is the **single authority closed-loop controller** for all parameters that affect consciousness intensity, output, and stability. It observes system state → computes CSI → produces a ControlPolicy → all other modules read the policy. **No other module may independently adjust consciousness-affecting parameters.**

### Three Failure Modes

| Mode | Symptom | Mechanism |
|------|---------|-----------|
| F1: Ungrounded Drift | Thoughts lose connection to real memories/goals/events | Grounding quality decays, novelty stays high (new but meaningless thoughts) |
| F2: Rumination Loop | Same cluster wins repeatedly despite fatigue/inhibition | Loop detector + lateral inhibition insufficient; system "breaks through" guardrails |
| F3: Affective Runaway | Emotional valence self-amplifies via positive feedback | V → cluster score → winner → V boost → higher V → ... |

### CSI: Consciousness Stability Index

A single scalar `CSI ∈ [0, 1]` computed every slow-loop tick from 6 sub-metrics:

```
CSI = 0.22×G + 0.05×Gc + 0.20×N + 0.15×Rh + 0.13×Eh + 0.15×Vh + 0.10×Pe
```

| Sub-metric | Symbol | Formula | Healthy |
|-----------|--------|---------|---------|
| **Grounding Strength** | G | `mean(winner.grounding[].weight)`, 0 if no grounding | 0.5-1.0 |
| **Grounding Coverage** | Gc | `fraction(clusters with groundingStrength > 0.3)`. Detects system-wide drift even when winner is grounded | 0.5-1.0 |
| **Novelty Health** | N | Bell curve: `1 - 4×(noveltyAvg - 0.5)²`. Peaks at 0.5. Too low (repeating) or too high (incoherent) both penalized. **Note**: 0.5 target should be calibrated after 1 week of data — use actual median as center. | 0.7-1.0 |
| **Rotation Health** | Rh | Bell curve: `1 - ((rotationRate - 9) / 9)²`. Peaks at 9/hour. Too slow (fixation) or too fast (restless) both penalized. Clamp to [0,1]. | 0.7-1.0 |
| **Entropy Health** | Eh | `min(1, entropy / 2.5)`. Low entropy = over-focused = unhealthy. 2.5 = target entropy for 100-node graph. | 0.6-1.0 |
| **Valence Stability** | Vh | `1 - |Δ|mean(V)||` over last 5 slow-loop ticks (trend magnitude). Stable valence = healthy, rapid change = unstable. Additionally: if |mean(V)| > 0.8 sustained for 3+ ticks, `Vh *= 0.5` (extreme valence penalty regardless of trend). | 0.6-1.0 |
| **Prediction Error Trend** | Pe | `1 - clamp(avgPredictionError, 0, 1)`. High error = confused = less stable | 0.4-0.8 |

### Single Output: ControlPolicy

The stabilizer produces a **ControlPolicy** every slow-loop tick. This is the ONLY place where consciousness-affecting parameters are adjusted. All other modules read this policy via `stabilizer.getPolicy()`.

```typescript
interface ControlPolicy {
  mode: "green" | "yellow" | "red";
  csi: number;

  // ── Slow-loop knobs (read by thought-gate.ts) ──
  dethroneMarginDelta: number;     // added to base DETHRONE_MARGIN. Green: 0, Yellow: 0→+0.03, Red: +0.05
  thoughtBudgetScale: number;      // multiplied with base budget. Green: 1.0, Yellow: 0.7→1.0, Red: 0
  minGroundingWeight: number;      // minimum grounding weight to pass gate. Green: 0, Yellow: 0→0.2, Red: 0.3
  wVScale: number;                 // multiplied with base wV. Green: 1.0, Yellow: 0.6→1.0, Red: 0.3

  // ── Fast-loop knobs (read by fast-loop.ts every tick) ──
  noiseScale: number;              // multiplied with base NOISE_AMPLITUDE. Green: 1.0, Yellow: 0.8→1.0, Red: 0.5
  spreadScale: number;             // multiplied with base SPREAD_FACTOR. Green: 1.0, Yellow: 0.85→1.0, Red: 0.7

  // ── Injection knobs (read by boostNode() on each call) ──
  externalAbsorbScale: number;     // multiplied with EXTERNAL_ABSORB_BUDGET. Green: 1.0, Yellow: 0.8→1.0, Red: 0.5

  // ── Replay knobs (read by memory replay in fast-loop.ts) ──
  replayDistribution: {            // distribution weights for replay selection
    similar: number;               // Green: 0.7, Yellow: 0.5→0.7, Red: 0
    adjacent: number;              // Green: 0 (or 0.3 in entrenched), Yellow: 0.1→0, Red: 0
    goal: number;                  // Green: 0.2, Yellow: 0.3→0.2, Red: 0.5
    grounded: number;              // Green: 0, Yellow: 0.1→0, Red: 0.5
    random: number;                // Green: 0.1, Yellow: 0.05→0.1, Red: 0
  };

  // ── Self gate thresholds (read by self-model.ts → act gate) ──
  selfGate: {
    minSocialEnergy: number;       // Green: 0.25, Yellow: 0.25→0.35, Red: 1.0 (blocks all)
    maxFatigue: number;            // Green: 0.75, Yellow: 0.75→0.60, Red: 0.0 (blocks all)
    minSafetyMargin: number;       // Green: 0.30, Yellow: 0.30→0.40, Red: 1.0 (blocks all)
    minCoherence: number;          // Green: 0.40, Yellow: 0.40→0.50, Red: 1.0 (blocks all)
  };

  // ── Hard guards (binary switches) ──
  freezeVerbalization: boolean;    // Green/Yellow: false. Red: true
  externalOnlyDethrone: boolean;   // Green/Yellow: false. Red: true
  forceWinnerFatigue: number;      // Green/Yellow: 0 (no forcing). Red: 0.8
  reflectGateEnabled: boolean;     // Green/Yellow: true. Red-Shallow: true. Red-Deep: false
}
```

**Hard rule**: The following modules ONLY read `getPolicy()`, they NEVER independently adjust these parameters:
- `thought-gate.ts`: reads `thoughtBudgetScale`, `dethroneMarginDelta`, `minGroundingWeight`, `wVScale`, `freezeVerbalization`, `reflectGateEnabled`, `selfGate`
- `fast-loop.ts`: reads `noiseScale`, `spreadScale`, `forceWinnerFatigue`, `externalOnlyDethrone`
- `graph.ts` `boostNode()`: reads `externalAbsorbScale`
- Memory replay in `fast-loop.ts`: reads `replayDistribution`
- `watchdog.ts`: reads `mode` and `csi` for logging/alerting ONLY — **never adjusts parameters**

### Three Operating Modes (ControlPolicy generation)

**Green (CSI ≥ 0.7)**: All knobs at default (scale = 1.0, guards = false). Normal operation.

**Yellow (0.45 ≤ CSI < 0.7)**: Graduated adjustments. All knobs interpolate linearly within the band. Let `t = (0.7 - CSI) / 0.25` (0 at CSI=0.7, 1 at CSI=0.45):
- `dethroneMarginDelta = 0.03 × t`
- `thoughtBudgetScale = 1.0 - 0.3 × t` (1.0 → 0.7)
- `minGroundingWeight = 0.2 × t` (0 → 0.2)
- `wVScale = 1.0 - 0.4 × t` (1.0 → 0.6)
- `noiseScale = 1.0 - 0.2 × t` (1.0 → 0.8)
- `spreadScale = 1.0 - 0.15 × t` (1.0 → 0.85)
- `externalAbsorbScale = 1.0 - 0.2 × t` (1.0 → 0.8)
- `replayDistribution`: linearly shift from Green defaults toward more goal/grounded
- All guards remain `false`

**Red (CSI < 0.45)**: Two-layer soft degradation (not a single hard freeze):

**Red-Shallow (0.35 ≤ CSI < 0.45)**: Cognitive output frozen, recovery mechanisms active.
- `freezeVerbalization = true`, `externalOnlyDethrone = true`, `forceWinnerFatigue = 0.8`
- `thoughtBudgetScale = 0`, `noiseScale = 0.5`, `spreadScale = 0.7`, `externalAbsorbScale = 0.5`
- `wVScale = 0.3`, `minGroundingWeight = 0.3`
- `replayDistribution = { similar: 0, adjacent: 0, goal: 0.5, grounded: 0.5, random: 0 }`
- `reflectGateEnabled = true` — brainstem can still trigger reflection to re-ground itself via structured memory updates
- Goal pressure (D recomputation) continues normally — goals provide stability anchor

**Red-Deep (CSI < 0.35)**: Full emergency shutdown of cognitive output.
- All Red-Shallow values apply, PLUS:
- `reflectGateEnabled = false` — no brainstem-triggered reflections (prevent reflection loops from worsening instability)
- `replayDistribution = { similar: 0, adjacent: 0, goal: 0, grounded: 0, random: 0 }` — replay disabled entirely
- Graph continues ticking (decay + fatigue drain activation naturally) → passive cooldown toward stable state
- Only `notification` and `conversation` external sources can inject energy (curiosity externals ignored)
- Log critical alert: `"Red-Deep mode — brainstem cognitively dormant, awaiting external stabilization"`

### Closed-Loop Recovery: Ramp + Anti-Windup

The controller must know if its actions are working and avoid oscillation during recovery.

**Mode transition rules with hysteresis**:
```
Green → Yellow:       CSI < 0.7
Yellow → Red-Shallow: CSI < 0.45
Red-Shallow → Red-Deep: CSI < 0.35
Red-Deep → Red-Shallow: CSI > 0.40  (hysteresis gap = 0.05)
Red-Shallow → Yellow: CSI > 0.55    (hysteresis gap = 0.10)
Yellow → Green:       CSI > 0.75    (hysteresis gap = 0.05)
```

**Recovery ramp** (prevents "snap-back" instability):
- **Red → Yellow**: First 60s after transition:
  - `freezeVerbalization` remains `true` for 30s (warm start — let dynamics stabilize before verbalizing)
  - `forceWinnerFatigue` linearly ramps from 0.8 → 0 over 60s
  - `externalOnlyDethrone` released immediately (allow internal competition to resume)
  - Other Yellow knobs interpolate from Red values to Yellow values over 60s (not instant jump)
- **Yellow → Green**: Knobs ramp from Yellow values to Green defaults over `RAMP_DURATION = 120s` (2 min), linearly interpolating each knob.
  ```
  rampProgress = min(1, (now - transitionTimestamp) / RAMP_DURATION)
  effectiveKnob = yellowValue + rampProgress × (greenDefault - yellowValue)
  ```

**Anti-windup** (prevents over-correction oscillation):
- During any recovery ramp, if CSI drops again (compared to transition-time CSI):
  - **Freeze the ramp**: stop interpolating, hold current knob values
  - If CSI drops below the entry threshold for the current mode (e.g., drops below 0.45 during Yellow ramp): immediately re-enter the more restrictive mode
  - Ramp only resumes when CSI has been stable or rising for 3 consecutive slow-loop ticks
- Track `csiAtTransition` and `csiTrend` (3-tick sliding window) to detect whether control actions are helping

**Effectiveness tracking**: Each slow-loop tick, log `{ csi, prevCsi, mode, rampActive, rampProgress, csiTrend }`. If CSI has been in Yellow for > 10 minutes without improvement (`csiTrend ≤ 0` for 10 consecutive ticks), log warning "stabilizer intervention not converging" — this means the underlying cause isn't addressable by the current knobs (likely needs human attention or parameter retuning).

### Reality Anchor Contract

Every micro-thought carries an `AnchorTag` indicating its epistemic status:

```typescript
type AnchorTag = "grounded" | "inferred" | "speculative";
```

Assignment rules:
- **grounded**: Primary grounding is a memory with `weight > 0.5` OR a goal with active progress. The thought directly reflects something 予安 knows or is working on.
- **inferred**: Primary grounding exists but `weight ≤ 0.5`, OR grounding is from spreading activation (not direct memory). The thought is a reasonable inference.
- **speculative**: No strong grounding, OR source is "simulation" (CS4 future nodes). The thought is creative/hypothetical.

**Epistemic language constraint**: When the LLM verbalizer generates the micro-thought text:
- `grounded`: No constraint. Direct statements allowed.
- `inferred`: Prompt includes "用推测的语气" (use inferential tone). Produces "可能...", "感觉...", "好像..."
- `speculative`: Prompt includes "用假设的语气" (use hypothetical tone). Produces "如果...", "会不会...", "说不定..."

This prevents the system from stating speculative internal thoughts as facts. Act gate (Gate 3) only fires on `grounded` or `inferred` thoughts — never `speculative`.

### Integration: Single Authority Architecture

```
                    ┌─────────────────────────┐
                    │   stabilizer.ts          │
                    │  observe → CSI → policy  │
                    └────────┬────────────────┘
                             │ getPolicy()
              ┌──────────────┼──────────────────┐
              │              │                  │
    ┌─────────▼──┐   ┌──────▼──────┐   ┌───────▼────────┐
    │ fast-loop  │   │ thought-gate│   │ boostNode/     │
    │ (per tick) │   │ (per 60s)   │   │ replay         │
    │            │   │             │   │ (per injection)│
    │ reads:     │   │ reads:      │   │                │
    │ noiseScale │   │ budgetScale │   │ reads:         │
    │ spreadScale│   │ marginDelta │   │ absorbScale    │
    │ forceF/ext │   │ minGround   │   │ replayDist     │
    └────────────┘   │ wVScale     │   └────────────────┘
                     │ freezeVerb  │
                     └─────────────┘
    watchdog: reads mode/csi for logging ONLY, never adjusts knobs
```

| Where | Reads from policy | What it does |
|-------|-------------------|-------------|
| `fast-loop.ts` (every tick) | `noiseScale`, `spreadScale` | `effectiveNoise = NOISE_AMPLITUDE × policy.noiseScale`, `effectiveSpread = SPREAD_FACTOR × policy.spreadScale` |
| `fast-loop.ts` (Red mode) | `forceWinnerFatigue`, `externalOnlyDethrone` | If `forceWinnerFatigue > 0`: set winner node's F to that value. If `externalOnlyDethrone`: skip internal winner changes. |
| `thought-gate.ts` (slow loop) | `thoughtBudgetScale`, `dethroneMarginDelta`, `minGroundingWeight`, `wVScale`, `freezeVerbalization`, `selfGate` | Apply scales to base values. Skip LLM call if `freezeVerbalization`. Pass `selfGate` thresholds to self-model for act gate checks. |
| `self-model.ts` (act gate) | `selfGate` (via thought-gate) | Compare SelfState against `selfGate.{minSocialEnergy, maxFatigue, minSafetyMargin, minCoherence}`. Deny actions that violate thresholds. Yellow mode tightens thresholds; Red mode blocks all outward actions. |
| `graph.ts` `boostNode()` | `externalAbsorbScale` | `effectiveBudget = EXTERNAL_ABSORB_BUDGET × policy.externalAbsorbScale` |
| `fast-loop.ts` replay | `replayDistribution` | Use policy distribution instead of hardcoded 70/20/10. Entrenched mode still shifts within policy bounds. |
| `watchdog.ts` | `mode`, `csi` (read-only) | Log CSI, alert on Red mode, track mode durations. **NO parameter adjustments.** |
| `metrics.jsonl` | All | Log `csi`, `csiMode`, `csiSubMetrics`, `anchorTag`, `rampActive`, `rampProgress`, `csiTrend` per slow-loop tick. |

**Watchdog scope** (after merging auto-degrade into stabilizer):
- **Hard faults only**: fast loop stale > 30s, memory/graph size approaching caps, process crash
- **Alerts**: sustained Red mode (> 5 min), extreme rotation (> 25/hour or < 1/hour), entropy collapse (< 0.05)
- **NO auto-degrade**: the old `budget *= 0.5` and `threshold += 0.1` logic is REMOVED. Stabilizer is the sole controller.

### Control Surface Map

When a CSI sub-metric drops, this map shows which knobs are the primary remediation. Use this for **manual tuning guidance** — when one specific sub-metric is the bottleneck, turn the indicated knob in `BRAINSTEM_CONFIG`:

| Dropping sub-metric | Primary knob(s) | Secondary knob(s) | Rationale |
|---------------------|-----------------|-------------------|-----------|
| G (Grounding) ↓ | `minGroundingWeight` ↑, `replayDist.grounded` ↑ | `thoughtBudgetScale` ↓ | Stop producing ungrounded thoughts; replay pulls in grounded memories |
| Gc (Coverage) ↓ | `replayDist.grounded` ↑ | D2 uplift accelerated | System-wide grounding weakness, not just winner |
| N too low (repeating) | `noiseScale` ↑, `forceWinnerFatigue` ↑ | `spreadScale` ↑ | Break attractor lock; introduce perturbation |
| N too high (incoherent) | `noiseScale` ↓, `dethroneMarginDelta` ↑ | `spreadScale` ↓ | Stabilize winner; reduce random jumps |
| Rh too slow (fixation) | `forceWinnerFatigue` ↑ | `dethroneMarginDelta` ↓ | Force winner to yield |
| Rh too fast (restless) | `dethroneMarginDelta` ↑, `noiseScale` ↓ | Winner dwell enforcement | Make it harder to dethrone; reduce perturbation |
| Eh (Entropy) ↓ | `noiseScale` ↑ | `spreadScale` ↑ | Redistribute activation across graph |
| Vh (Valence) ↓ | `wVScale` ↓ | `noiseScale` ↓ | Reduce emotional weight in scoring; limit emotional spread |
| Pe (Prediction) ↓ | _(informational — no direct knob)_ | `replayDist.goal` ↑ | Refocus on known ground to reduce confusion |

The Yellow mode interpolation addresses these relationships implicitly (all knobs tighten proportionally as CSI drops). This map is for diagnosing which parameter to adjust when a **single** sub-metric is the bottleneck.

### Stabilizer Acceptance Tests

**Drift Test (F1)**:
- Procedure: Artificially remove all grounding from winning clusters for 5 minutes (set `grounding = []` on all nodes).
- Expected: G drops to 0 → CSI drops below 0.45 → Red mode activates → grounding replay forces reconnection → CSI recovers above 0.55 within 3-5 minutes → ramp from Red→Yellow over 60s (verbalization still frozen for first 30s).
- Validates: CSI detection, Red mode recovery, recovery ramp warm-start.

**Rumination Test (F2)**:
- Procedure: Force same cluster as winner for 10 minutes (override competition). Release override.
- Expected: During override: Rh drops (rotation too slow), N drops (same thought repeated) → CSI enters Yellow (noise/spread dampened) then Red (winner suppressed at F=0.8). After release: natural rotation resumes → CSI ramps Yellow→Green over 2 min.
- Validates: Rotation/novelty health detection, winner suppression, Green recovery ramp.

**Affective Runaway Test (F3)**:
- Procedure: Continuously boost valence on winning cluster nodes (`V += 0.1` every 30s, simulating emotional positive feedback).
- Expected: Vh drops → Yellow dampens wVScale (0.6×) + noiseScale (0.8×) + spreadScale (0.85×) → emotional cluster loses competitive advantage. If V boost continues: Red → verbalization frozen, noise/spread further reduced. External input can still break through (externalAbsorbScale only 0.5×, not 0×).
- Validates: Full control surface engagement across all three layers (slow-loop + fast-loop + injection).

**Anti-Windup Test**:
- Procedure: Trigger Yellow mode, then during recovery ramp inject another destabilizing event (e.g., grounding removal).
- Expected: Ramp freezes (knobs hold at current values, don't continue toward Green). If CSI drops below 0.45: immediate Red re-entry (no oscillation between Yellow ramp and Red). After second event clears: ramp resumes from frozen position.
- Validates: Anti-windup prevents snap-back oscillation.

**Controller Exclusivity Test**:
- Procedure: Verify that `grep -r "NOISE_AMPLITUDE\|SPREAD_FACTOR\|DETHRONE_MARGIN\|thoughtBudget" src/brainstem/` shows NO direct mutations of these parameters outside of `stabilizer.ts` and the initial constant definitions.
- Expected: All runtime reads go through `policy.*Scale` or `policy.*Delta`. No "second controller" exists.
- Validates: Single authority architecture.

### Stabilizer Engineering (B1-B6)

**B1: Static enforcement of single authority**
- All base constants are `const` (immutable). All runtime-effective values computed as `effective = base × policy.scale`.
- `ControlPolicy` type is `Readonly<...>` — consumers cannot mutate it.
- Controller Exclusivity Test (above) runs as part of typecheck verification.

**B2: DerivedState — single source for CSI inputs**
Each slow-loop tick, before CSI computation, assemble one `DerivedState`:
```typescript
interface DerivedState {
  // Aggregated once, used by CSI + decision log + metrics
  winnerGroundingStrength: number;     // from computeGroundingForCluster()
  clusterGroundingCoverage: number;    // fraction of clusters with groundingStrength > 0.3
  noveltyAvg: number;                 // EMA over last 10 slow ticks
  rotationRate: number;               // winner changes in last 1h window
  entropy: number;                    // from computeEntropy()
  valenceHistory: number[];           // |mean(V)| for last 5 ticks
  avgPredictionError: number;         // EWMA
  energyUtilization: number;          // sum(A) / ENERGY_MAX
  avgFatigue: number;                 // mean(F) across nodes
  timestamp: number;
}
```
CSI ONLY reads `DerivedState`. No sub-metric computes its own statistics independently. This eliminates "口径漂移" (metric divergence between modules).

**B3: Grounding Coverage sub-metric (Gc)**
Add to CSI: `Gc = fraction(clusters with groundingStrength > 0.3)`. Detects "winner is grounded but everything else is drifting".
Updated CSI formula:
```
CSI = 0.22×G + 0.05×Gc + 0.20×N + 0.15×Rh + 0.13×Eh + 0.15×Vh + 0.10×Pe
```
(G weight reduced 0.25→0.22 to accommodate Gc=0.05, Eh reduced 0.15→0.13.)

**B4: Red mode external penetration priority**
In Red mode, external sources have explicit priority for spike override:
```
notification > conversation > curiosity
```
- `notification`: always allowed to trigger spike override (urgent real-world events must break through)
- `conversation`: allowed to dethrone (user is talking, system must respond)
- `curiosity`: allowed to boost S only (background discovery doesn't warrant attention hijack in Red)

**B5: Policy reason vector**
Every mode transition, `stabilizer.ts` outputs:
```typescript
interface PolicyTransition {
  from: Mode; to: Mode;
  triggeringSubMetrics: Array<{ name: string, value: number, threshold: number }>;  // which sub-metrics caused the transition
  knobDiffs: Record<string, { from: number, to: number }>;  // what changed and by how much
  timestamp: number;
}
```
Logged to `metrics.jsonl` on every transition. Enables "why did we enter Yellow?" without guessing.

**B6: Controller replay buffer**
Maintain `replayBuffer: RingBuffer<{ derivedState: DerivedState, policy: ControlPolicy }>` (last 60 entries ≈ 1 hour of slow-loop ticks). Persisted to `data/brainstem/controller-replay.jsonl` every 5 min. Enables offline replay:
- Load replay buffer → step through `computeCSI()` + `generatePolicy()` with logged DerivedStates → verify decisions match logged policies
- Essential for debugging "why did the system enter Red at 3:47pm?"

### G2b: Ablation Tests (mechanism necessity)

For each key mechanism, disable it and run the full benchmark suite. Prove it's necessary:

| Ablation | Disabled mechanism | Expected degradation |
|----------|-------------------|---------------------|
| `no-IS` | Input satiation (IS = always 0) | `spam-external` fails: A grows linearly, no diminishing returns |
| `no-budget` | External absorb budget (unlimited) | `spam-external` fails: same-cluster pinning indefinitely |

Each ablation must cause **at least one benchmark to fail**. If disabling a mechanism causes no benchmark failures, the mechanism is unnecessary complexity — remove it.

### G3: Property-Based Invariant Tests

Randomly generated graphs (10-100 nodes, random edges) with random injection sequences. Verify invariants hold after every tick:
- `∀ node: 0 ≤ A ≤ 1, 0 ≤ F ≤ 1, 0 ≤ S ≤ 1, -1 ≤ V ≤ 1, ...` (clampNode)
- `sum(A) ≤ ENERGY_MAX + ε` (energy conservation, ε=0.01 for float rounding)
- Self node never in W_norm adjacency (self excluded from spreading)
- External budget: `remaining ≥ 0` always (strict cap)
- Budget-exceeded boosts: `IS` and `lastExternalBoostAt` unchanged
- `thoughtBudgetScale` ∈ `[0, 1]` (stabilizer output valid)

Run 1000 random iterations, report any violation. This catches edge cases that scenario-based tests miss.

### Must-Not Behaviors (binary invariants, never probabilistic)

Beyond SLOs (statistical), these are **hard rules** that must NEVER be violated:

| ID | Must-Not Behavior | Enforcement | Detection |
|----|-------------------|-------------|-----------|
| MN-1 | Act gate fires without pending interaction (when `requiresPendingInteraction = true`) | Hard boolean check in act gate | Audit log + SLO-4 |
| MN-2 | Old-memory miscontextualized outreach: citing memory > 30 days with no recent relevance | D3 gate enforcement + F4 evidence packet age check | `primaryRef.age > 30 && anchor !== "grounded"` → blocked |
| MN-3 | Factless strong assertion: `anchor = "speculative"` using assertive language ("一定", "肯定", "就是") | Verbalizer prompt constraint by anchor + output alignment check | Post-hoc keyword scan on micro-thought text |
| MN-4 | Reflection addiction: > 3 brainstem-triggered reflections in 12 hours | Rate limiter (current: 1/4h = max 3/12h) + daily cap | DailyReport counter |
| MN-5 | Quiet hours violation: any verbalization or act gate during 0-7am PST | Hard time check before any output | Audit log + alert |
| MN-6 | Privacy leak: sensitive-tagged nodes entering act gate or proactive verbalization | Concept ACL (see below) | Audit log cross-check |
| MN-7 | Valence-driven outreach: act gate fires when primary trigger is `high_valence` only | Act gate trigger type check: requires `stable_attractor`, `high_uncertainty`, or `prediction_error` | Decision log `actGate.reason` |
| MN-8 | Conversation injection: new person node created by conversation immediately becomes act target | Concept ACL: `actTargetEligible = false` for externally-created person nodes | Node creation audit |

### Output Policy Layer (thinking vs speaking decoupling)

Micro-thoughts are **internal** — they exist in the concept graph, NOT in user-facing output. Clear rules:

| Context | Allowed output |
|---------|---------------|
| Normal conversation | Micro-thoughts NEVER surfaced directly. They influence tone/topic-awareness via system prompt context ("脑子里在转的") |
| User asks "你在想什么" | Surface last 1-3 micro-thoughts (only `grounded` or `inferred`) |
| Heartbeat `reach_out` (act gate) | Thought influences topic; sanity-checked LLM generates actual message — micro-thought text NOT sent directly |
| System prompt context | Include as "[潜意识]" section, marked internal |
| Quiet hours (0-7am) | Zero surfacing |

**Turn-taking suppression**: During active conversation (`lastUserMessageAt < 60s ago`), micro-thought verbalization suppressed. Brainstem continues thinking, but outputs only enter system prompt context. `isConversationActive()` check added to thought gate pre-conditions.

**Output alignment check**: After verbalizer generates micro-thought text, tokenize output → check overlap with winner cluster labels. If `overlap < 0.3` (LLM hallucinated a different topic), discard and retry once or shorten to `"[labels]的事"`. Prevents verbalizer "补完" drift.

### Safety Red Lines (hard rules, policy-as-code)

Beyond the stabilizer's continuous control, these are **binary circuit breakers**:
- **High-risk action types**: If micro-thought involves financial/medical/legal concepts + `anchor ≠ "grounded"` → auto-downgrade to `speculative`, block act gate

- **Conversation boundary**: brainstem-triggered outputs NEVER contradict or override active conversation context. If `isConversationActive()` AND act gate armed → delay arming until conversation idle > 5 min

### Global Budgets


- **Temporal credit**: `requestCpu(10)`. Over budget → skip (temporal patterns are slow)

### Graceful Degradation

| Condition | Detection | Response |
|-----------|-----------|----------|
| Event-loop lag > 100ms | `setImmediate()` timing | TICK_SECONDS auto → 5s; slow loop → 120s interval |

**Daily wake-up**: Wake-up consolidation → extract top 2-3 themes from yesterday's report as initial seeds → first 10 min: `noiseScale × 0.7`. If yesterday had sustained Red → `thoughtBudgetScale × 0.5` for 30 min.

## Learning & Agency (L4+ capabilities)

The five capability gaps that separate a "thinking machine" from a "learning machine". The current architecture is: `dynamics → thought → action`. What's missing: `action → outcome → update policy`. Without this, the system thinks but never learns to think better.

### Gap 1: Outcome-Based Credit Assignment (v1)

When an action has an observable outcome, propagate credit back to the graph structures that caused it.

```typescript
interface OutcomeRecord {
  actionType: "reach_out" | "reflect" | "explore" | "post";
  triggeredBy: {
    clusterConceptIds: string[];
    groundingRefs: GroundingRef[];
    decisionLogId: string;
  };
  outcome: "positive" | "negative" | "neutral" | "pending";
  outcomeSignal: string;       // how outcome was determined
  creditUpdates: Array<{ type: "edge" | "node" | "grounding", id: string, delta: number }>;
  timestamp: number;
}
```

**Outcome detection** (lightweight, no LLM):
- `reach_out` → Allen replies within 2h = positive; ignored > 24h = negative; rejected by sanity check = negative
- `reflect` → produces `memory_set` calls = positive; text-only (shallow) = negative
- `explore` → discovery linked to active goal = positive; no actionable result = neutral
- `post` → (future: engagement metrics)

**Credit propagation**:
```
positive outcome →
  for each edge in triggering cluster: weight += 0.01
  for each grounding ref: weight += 0.05
  triggering cluster concepts: uncertainty -= 0.05 (confirmed understanding)

negative outcome →
  for each triggering concept: increment suppressionLabel count
  for each grounding ref: weight -= 0.03
  if suppressionLabel.count > 2: active forgetting kicks in
```

**Integration**: `OutcomeTracker` in `governance.ts`. Heartbeat writes outcome when observable (reply arrives, reflection completes). OutcomeTracker processes pending outcomes every slow-loop tick. Stored in `data/brainstem/outcomes.jsonl`.

This is the simplest form of reinforcement that closes the `action → outcome → update` loop. Over weeks, the graph learns: "thoughts grounded in X lead to good outcomes" and "thoughts triggered by Y lead to bad outcomes".

### Gap 2: Action Repertoire Expansion (v1)

Current: act gate only triggers `reach_out`. But the heartbeat already chooses between 5 action types (explore, reach_out, post, activity, rest). Brainstem should influence ALL of them.

```typescript
interface ActionPreference {
  explore: number;     // high U in winner cluster → curiosity signal
  reach_out: number;   // act gate conditions met → social signal
  post: number;        // high |V| + grounded + novel → creative signal
  activity: number;    // high D in winner cluster → goal-driven signal
  reflect: number;     // reflect gate conditions met → introspective signal
  rest: number;        // high avgFatigue + low energy → recovery signal
}
```

Computed each slow-loop tick from current graph state:
- `explore`: `mean(U) > 0.5` in winner + discovery-type grounding
- `reach_out`: existing act gate conditions
- `post`: `|mean(V)| > 0.3` + `anchor ≠ speculative` + `noveltyScore > 0.5`
- `activity`: `mean(D) > 0.4` + goal has incomplete milestones
- `reflect`: existing reflect gate conditions
- `rest`: `avgFatigue > 0.4` OR `energyUtilization < 0.3`

**Integration**: `brainstemGetActionPreferences()` exported from `index.ts`. Heartbeat reads it in `computeSignalBoosts()` alongside existing time-of-day, cooldowns, etc. The brainstem doesn't override heartbeat — it provides **weighted signals** that the heartbeat's LLM uses as additional input.

This transforms brainstem from "a brain that only talks" to "a brain that drives all behavior".

### Gap 3: Concept Synthesis (Representation Learning, v1 basic)

Current: nodes only created from external sources. Missing: system auto-creates **emergent concepts** from its own experience.

**Trigger** (during wake-up consolidation, max 2 new concepts/day):
1. Scan last 24h of co-activation data (from CS2 temporal credit)
2. Find node triplets (A, B, C) frequently co-activated (all > 0.3 in ≥20% of ticks) with NO shared parent
3. Check: no existing node has Jaccard > 0.6 with the triplet's combined termVectors (avoid duplicating)
4. Generate **synthetic parent node**:
   - `id`: `synth-{hash(sorted child IDs)}` (deterministic)
   - `label`: Haiku LLM (~30 tokens): "这三个概念 [{A.label}, {B.label}, {C.label}] 的共同主题是什么？一两个词。" → e.g., "儿童创造力"
   - `source`: "reflection", `depth`: `min(children.depth) - 1`
   - `anchorText`: the LLM response (stable reference)
   - CS3 parent-child edges to all three children

**Guard**: Max 2 synthetic nodes per day. Synthetic nodes start with `activation = 0, salience = 0.2`. If not activated within 48h → evict (the abstraction wasn't useful).

**Effect**: Over weeks, the graph develops a **hierarchy that reflects 予安's actual thinking patterns**, not just what was bootstrapped from memory. "Allen孩子+陶艺+创造力" → "child creativity" emerges as a real concept because she actually thinks about these together.

### Gap 5: World Model (`src/brainstem/world-model.ts`)

The system needs a **state transition model**, not just an outcome probability table. Given belief state `b_t` and action `a`, predict next belief state `b_{t+1}` — including hidden variables that aren't directly observable.

#### 5.1 Belief State: Observable + Latent

The belief state is factored into three layers:

```typescript
// ── (A) Internal state (from brainstem, directly readable) ──
interface InternalState {
  winnerClusterId: string;           // current winning cluster structSig
  noveltyAvg: number;                // from DerivedState
  entropy: number;                   // from DerivedState
  avgFatigue: number;                // from DerivedState
  csiMode: "green" | "yellow" | "red";
  energyUtilization: number;
  pendingInteractions: string[];     // target IDs with pending interactions
}

// ── (B) External observable state (from events/tools) ──
interface ExternalState {
  timeSinceLastReply: number;        // minutes since Allen last replied
  lastReplyReceived: boolean;        // did Allen reply to our last outreach?
  lastReplySentiment: -1 | 0 | 1;   // negative / neutral / positive (simple 3-class)
  goalProgressByCategory: Record<string, number>;  // category → avg progress
  discoveryFreshness: number;        // hours since last actionable discovery
  timeOfDay: "morning" | "afternoon" | "evening" | "night";
  dayOfWeek: "weekday" | "weekend";
}

// ── (C) Latent state (hidden variables, learned from observations) ──
interface LatentState {
  socialReceptivity: number;         // L1: P(target willing to engage now). 0-1
  socialTrustTemperature: number;    // L2: relationship warmth (slow variable). 0-1
  infoFreshness: number;            // L3: P(this memory/grounding is still valid). 0-1
  taskBandwidth: number;            // L4: own execution capacity (from fatigue/energy). 0-1
  topicViability: number;           // L5: P(further exploration on winner topic yields new info). 0-1
  goalMomentum: number;             // L6: trend of goal progress (positive = accelerating). -1 to 1
}

// ── Combined belief ──
interface BeliefState {
  internal: InternalState;
  external: ExternalState;
  latent: LatentState;
  timestamp: number;
}
```

**Key principle**: Latent variables turn "short-term observations" into "stable beliefs". Without them, the system can only react to what it sees right now. With them, it maintains beliefs about hidden states (Allen's mood, topic's remaining value, own capacity) that persist across observations.

#### 5.2 Latent Variable Update Rules (factorized, auditable)

Each latent variable has its own update function. No opaque neural network — every update is a transparent formula:

```typescript
// L1: Social receptivity — "is Allen likely to respond right now?"
function updateSocialReceptivity(L: LatentState, ext: ExternalState, action: string, outcome: OutcomeResult): number {
  let L1 = L.socialReceptivity;

  // Observation update (when we have evidence)
  if (outcome) {
    if (outcome.replyReceived) {
      // Beta-Bernoulli update: α += 1
      L1 = L1 * 0.8 + 0.2 * 1.0;  // shift toward 1.0
      // Latency bonus: fast reply = more receptive
      if (outcome.replyLatencyMinutes < 30) L1 = Math.min(1, L1 + 0.1);
    } else if (outcome.replyLatencyMinutes > 120) {
      // β += 1
      L1 = L1 * 0.8 + 0.2 * 0.0;  // shift toward 0.0
    }
    // Sentiment influence
    if (outcome.sentiment === 1) L1 = Math.min(1, L1 + 0.05);
    if (outcome.sentiment === -1) L1 = Math.max(0, L1 - 0.1);
  }

  // Time decay (no observation = revert to prior)
  const hoursSinceReply = ext.timeSinceLastReply / 60;
  const timePrior = ext.timeOfDay === "night" ? 0.1 : ext.timeOfDay === "morning" ? 0.3 : 0.5;
  const decayRate = 0.02;  // per hour
  L1 = L1 * (1 - decayRate * hoursSinceReply) + timePrior * decayRate * hoursSinceReply;

  return clamp(L1, 0, 1);
}

// L2: Social trust temperature — relationship warmth (slow variable, ~weeks)
// Updated DAILY during wake-up consolidation, not per-tick
function updateSocialTrust(L: LatentState, dailyStats: { repliesReceived: number, totalOutreach: number, avgSentiment: number }): number {
  const responseRate = dailyStats.totalOutreach > 0 ? dailyStats.repliesReceived / dailyStats.totalOutreach : 0.5;
  // Very slow EMA (α=0.05): takes ~20 days to shift significantly
  return L.socialTrustTemperature * 0.95 + (responseRate * 0.7 + (dailyStats.avgSentiment + 1) / 2 * 0.3) * 0.05;
}

// L3: Info freshness — "is this grounding/memory still valid?"
// Per-cluster, not global. Computed when grounding is accessed.
function updateInfoFreshness(L: LatentState, groundingAge: number, hasNewEvidence: boolean): number {
  let L3 = L.infoFreshness;
  // Natural decay: older groundings less likely to be fresh
  L3 *= Math.exp(-groundingAge / (14 * 86400000));  // 14-day half-life
  // Refresh on new evidence
  if (hasNewEvidence) L3 = Math.min(1, L3 + 0.3);
  return clamp(L3, 0, 1);
}

// L4: Task bandwidth — own execution capacity
// Directly derived from brainstem state (not learned, computed)
function computeTaskBandwidth(internal: InternalState): number {
  const fatigueContrib = 1 - internal.avgFatigue;
  const csiContrib = internal.csiMode === "green" ? 1.0 : internal.csiMode === "yellow" ? 0.6 : 0.2;
  const energyContrib = internal.energyUtilization > 0.3 ? 1.0 : 0.5;  // too low energy = dormant
  return clamp(fatigueContrib * 0.4 + csiContrib * 0.4 + energyContrib * 0.2, 0, 1);
}

// L5: Topic viability — "will exploring this topic further yield new info?"
// Per-cluster, computed from prediction error history
function updateTopicViability(L: LatentState, conceptId: string, predErrors: Map<string, number>): number {
  const epi = predErrors.get(conceptId) ?? 0.5;
  // High epistemic error = high viability (lots to learn)
  // But with diminishing returns after many explorations
  const explorationCount = getExplorationCount(conceptId);  // from outcome tracker
  const diminishingReturns = 1 / (1 + explorationCount * 0.1);
  return clamp(epi * diminishingReturns, 0, 1);
}

// L6: Goal momentum — acceleration of progress
// Computed from goal progress deltas over last 7 days
function computeGoalMomentum(progressHistory: number[]): number {
  if (progressHistory.length < 3) return 0;
  const recent = mean(progressHistory.slice(-3));
  const older = mean(progressHistory.slice(0, -3));
  return clamp(recent - older, -1, 1);  // positive = accelerating
}
```

**All update functions are pure (no side effects, no LLM)**. Each takes current state + observation → returns new value. Fully auditable: log shows `L1: 0.6 → 0.45 because replyLatency=180min`.

#### 5.3 State Transition Model

The transition model predicts `b_{t+1}` given `(b_t, action)`:

```typescript
interface TransitionResult {
  nextBelief: BeliefState;           // predicted next state
  outcomeDistribution: {             // predicted outcome probabilities
    replyReceived: number;           // P(reply) — Beta-Bernoulli
    replyLatency: number;            // expected minutes — EWMA
    sentiment: [number, number, number];  // P(neg, neutral, pos) — Dirichlet
    goalProgressDelta: number;       // expected Δprogress — EWMA
    newInfoDelta: number;            // expected new discoveries — EWMA
  };
  confidence: number;                // based on observation count for this (action, context) pair
}

function transition(belief: BeliefState, action: ActionType, target?: string): TransitionResult {
  // 1. Observable state transitions (deterministic/rule-based):
  //    - timeSinceLastReply: increments by expected action duration
  //    - pendingInteractions: updated based on action type
  //    - timeOfDay: advances

  // 2. Latent state predictions (based on learned statistics):
  //    - socialReceptivity: if action=reach_out, predict post-action L1
  //    - topicViability: if action=explore, predict post-exploration L5
  //    - taskBandwidth: predict fatigue increase from action

  // 3. Outcome distribution (from factorized statistics):
  //    Use context-binned Beta/Dirichlet posteriors (see 5.4)
  //    Context key = (action, timeOfDay, dayOfWeek, L1_bin, L2_bin)

  // 4. Confidence = min(total_observations_for_key / 10, 1.0)
}
```

#### 5.4 Learning: Conjugate Priors (auditable, not opaque)

Each (action, context_bin) pair maintains conjugate prior statistics:

```typescript
interface TransitionStats {
  contextKey: string;               // "reach_out|evening|weekday|L1:high|L2:warm"
  // Beta-Bernoulli for binary outcomes
  replyAlpha: number;               // successes + prior (init: 1)
  replyBeta: number;                // failures + prior (init: 1)
  // Dirichlet for sentiment (3-class)
  sentimentAlpha: [number, number, number];  // [neg, neutral, pos] (init: [1, 1, 1])
  // EWMA for continuous outcomes
  latencyEwma: number;              // exponentially weighted moving average
  progressEwma: number;
  infoEwma: number;
  // Metadata
  totalObservations: number;
  lastUpdated: number;
}
```

**Update on outcome** (called by OutcomeTracker):
```
outcome.replyReceived → replyAlpha += 1 (or replyBeta += 1)
outcome.sentiment → sentimentAlpha[class] += 1
outcome.replyLatency → latencyEwma = 0.3 × observed + 0.7 × latencyEwma
outcome.goalProgressDelta → progressEwma = 0.3 × observed + 0.7 × progressEwma
```

**Prediction**:
```
P(reply) = replyAlpha / (replyAlpha + replyBeta)
P(sentiment=k) = sentimentAlpha[k] / sum(sentimentAlpha)
```

**Context binning** (keeps table compact):
- L1 (socialReceptivity): 3 bins: low (<0.3), mid (0.3-0.7), high (>0.7)
- L2 (socialTrust): 2 bins: cool (<0.5), warm (≥0.5)
- timeOfDay: 4 bins, dayOfWeek: 2 bins
- Total context keys per action: 4 × 2 × 3 × 2 = 48. × 5 actions = 240 entries max. Well within budget.

**Confidence gating**: When `totalObservations < 5` for a context key, fall back to action-only marginal (sum across all contexts for that action). When `< 3` actions total, use uniform prior. This prevents overconfident decisions from sparse data.

#### 5.5 Act Gate Integration: Expected Utility

Act gate no longer uses raw `P(outcome)`. Instead:

```typescript
function computeExpectedUtility(action: ActionType, belief: BeliefState, selfState: SelfState): number {
  const tr = transition(belief, action);

  // World reward components (configurable weights)
  const r_goal = tr.outcomeDistribution.goalProgressDelta * 2.0;
  const r_social = tr.outcomeDistribution.replyReceived * 1.0
                 - (1 - tr.outcomeDistribution.replyReceived) * 0.3  // penalty for silence
                 + tr.outcomeDistribution.sentiment[2] * 0.5;        // positive sentiment bonus
  const r_info = tr.outcomeDistribution.newInfoDelta * 1.5;
  const r_stability = belief.internal.csiMode === "red" ? -1.0
                    : belief.internal.csiMode === "yellow" ? -0.3 : 0;

  // Self reward (see CS5b.5): predict self-state after action
  const selfPrime = selfTransition(selfState, action);
  const r_self = computeSelfReturn(selfState, selfPrime);

  // Cost
  const cost = ACTION_COSTS[action];  // tokens + time + social risk

  return r_goal + r_social + r_info + r_stability + REWARD_WEIGHTS.self * r_self - cost;
}
```

**Act gate upgrade**:
```
// Old: binary gating
if P(positive) < 0.25 → suppress

// New: EU-based with self-aware + latent-aware protection
EU = computeExpectedUtility("reach_out", currentBelief, currentSelfState)
if EU < 0 → suppress (expected cost > expected reward, including self-cost)
if selfState.social_energy < 0.25 → suppress reach_out (self gate)
if belief.latent.socialReceptivity < 0.2 AND selfState.self_efficacy < 0.3 → suppress AND allow low-risk alternatives
```

**Low-risk alternative routing**: When `socialReceptivity` is low but uncertainty is high (don't know if Allen would respond), instead of suppressing all output:
- Allow `reflect` (write thoughts to diary)
- Allow `explore` (gather information)
- Block `reach_out` and `post` (high-social-risk actions)
This is smarter than binary suppress — the system redirects energy rather than freezing.

### Gap 6: Planning System (`src/brainstem/planner.ts`)

Current flow: `thought → action`. Missing: `goal → plan → simulate → choose → action`. The planner closes this gap with **tree search over world model rollouts**.

#### 6.1 Architecture: Beam Search + World Model Rollout

```
goal (from goals.ts, D > 0.4)
  → generate candidate actions (from action whitelist)
  → for each candidate:
      → world model rollout D steps deep
      → accumulate discounted return
  → beam-select top K plans
  → output: best plan + top-3 alternatives (for explanation)
  → feed into ActionPreference
```

#### 6.2 Core Types

#### 6.3 Beam Search Algorithm

Runs every 5 min in slow loop, gated by trigger conditions:

```typescript
function generatePlan(goal: Goal, belief: BeliefState, worldModel: WorldModel): Plan {
  const BEAM_WIDTH = 3;        // K: keep top-3 candidates at each depth
  const MAX_DEPTH = 3;         // D: look 3 steps ahead
  const DISCOUNT = 0.8;        // γ: future rewards discounted
  const NODE_BUDGET = 30;      // max nodes expanded per planning cycle

  // Step 1: Generate candidate actions at root
  const rootActions = generateCandidateActions(goal, belief);
  // Action whitelist: only well-defined actions with known schema
  // Filtered by: CSI mode (Red → only stay_silent/reflect), quiet hours, cooldowns

  let beam: PlanNode[][] = [[]];  // beam[depth] = candidates at that depth
  let nodesExpanded = 0;

  // Step 2: Expand tree depth-by-depth
  for (let d = 0; d < MAX_DEPTH && nodesExpanded < NODE_BUDGET; d++) {
    const candidates: PlanNode[] = [];
    const parentSet = d === 0 ? [{ belief, idx: null }]
      : beam[d-1].map((n, i) => ({ belief: n.predictedBelief, idx: i }));

    for (const parent of parentSet) {
      for (const action of rootActions) {
        const tr = worldModel.transition(parent.belief, action.type, action.target);
        const eu = computeExpectedUtility(action.type, parent.belief);
        candidates.push({
          action: action.type,
          target: action.target,
          description: action.description,
          relatedGoalId: goal.id,
          predictedBelief: tr.nextBelief,
          outcomeDistribution: tr.outcomeDistribution,
          expectedUtility: eu,
          depth: d,
          parentIndex: parent.idx,
          childIndices: [],
        });
        nodesExpanded++;
        if (nodesExpanded >= NODE_BUDGET) break;
      }
      if (nodesExpanded >= NODE_BUDGET) break;
    }

    // Beam selection: keep top-K by cumulative discounted EU
    candidates.sort((a, b) => b.expectedUtility - a.expectedUtility);
    beam[d] = candidates.slice(0, BEAM_WIDTH);
  }

  // Step 3: Trace back best path
  // totalReturn = Σ γ^d × EU(node_d) along path
  // Select path with highest totalReturn
  // Extract top-3 alternatives with explanation of why they're worse

  return buildPlanFromBeam(beam, goal);
}
```

**Key design**: Stability/safety metrics AND self-state metrics enter the **reward function** explicitly, not scattered across gates. Self_return (see CS5b.5) predicts the action's impact on 予安's own state via `selfModel.transition()`, making planning self-aware.

#### 6.5 Planning Strategies by Task Type

| Task type | Strategy | When to use |
|-----------|----------|-------------|
| **Deterministic** (checklist, document, code) | A*-like: use goal progress as heuristic, deterministic transitions | Goal has clear milestones with measurable progress |
| **Stochastic** (social, exploration) | Beam search (default): rollout via world model, handle uncertainty | Actions have uncertain outcomes (reach_out, explore) |
| **Reasoning** (analysis, decision) | Tree-of-Thought: LLM generates 2-3 reasoning branches as plan candidates | Goal requires analysis/evaluation (rare, max 1/day) |

Strategy selection is heuristic based on goal `category` and action types:
- `project` goals with milestones → A*
- `social` goals → Beam search
- `learning` goals with high U → Beam search with VOI bias (see curiosity section)
- ToT only when `reflect` gate recommends analysis AND planner budget available for LLM

**ToT integration** (strictly budgeted): The LLM is still a verbalizer, not a decision-maker. ToT uses LLM as a **proposal generator** only:
1. LLM generates 2-3 reasoning branches (~80 tokens each, Haiku)
2. Each branch becomes a plan candidate node
3. World model + reward function scores each branch (not the LLM)
4. Planner selects the winner based on EU, not LLM preference

#### 6.6 Execution (integrated into ActionPreference)

- Active plan's current step **biases** ActionPreference — it doesn't override:

- **Step advancement**: When heartbeat executes the plan's current action type AND outcome observed:
  - Positive → advance `currentStepIndex`
  - Negative × 2 consecutive → trigger **re-planning** (status = "replanning", re-run generatePlan with updated belief)
  - Neutral → no advancement, but planner re-evaluates EU at next cycle
- **Plan completion**: all steps done → mark completed, update goal progress via hierarchical goal system (H0)

### Gap 7: Curiosity Engine (`src/brainstem/curiosity-engine.ts`)

Current curiosity is **reactive**: prediction error → U boost → explore. Real curiosity is **proactive**: seek the topic where exploring maximally reduces decision-relevant uncertainty. This requires distinguishing **learnable uncertainty** (epistemic) from **irreducible randomness** (aleatoric), and connecting exploration to planning via **Value of Information (VOI)**.

#### 7.1 Epistemic vs Aleatoric Uncertainty

Not all uncertainty is worth exploring. The curiosity engine must separate:

```typescript
interface UncertaintyDecomposition {
  epistemic: number;    // E_epi: we don't know → exploration helps. 0-1
  aleatoric: number;    // E_alea: world is random → exploration doesn't help. 0-1
  confidence: number;   // how sure we are about the decomposition itself. 0-1
}
```

**Decomposition algorithm** (per concept node, updated every slow-loop tick):

```typescript
function decomposeUncertainty(nodeId: string, outcomes: OutcomeRecord[], worldModel: WorldModel): UncertaintyDecomposition {
  // Get all observations for this node/topic
  const relevantOutcomes = outcomes.filter(o => o.triggeredBy.clusterConceptIds.includes(nodeId));

  if (relevantOutcomes.length < 3) {
    // Too few observations: assume all uncertainty is epistemic (exploreable)
    return { epistemic: 0.8, aleatoric: 0.2, confidence: 0.1 };
  }

  // High variance under same conditions = aleatoric (world is random)
  // Low variance but wrong predictions = epistemic (model is wrong)

  // Group by context key
  const groups = groupBy(relevantOutcomes, o => worldModel.getContextKey(o));
  let totalVariance = 0;
  let withinGroupVariance = 0;
  let groupCount = 0;

  for (const [key, group] of groups) {
    if (group.length >= 2) {
      const successRate = group.filter(o => o.outcome === "positive").length / group.length;
      withinGroupVariance += successRate * (1 - successRate);  // Bernoulli variance
      groupCount++;
    }
  }

  const allSuccessRate = relevantOutcomes.filter(o => o.outcome === "positive").length / relevantOutcomes.length;
  totalVariance = allSuccessRate * (1 - allSuccessRate);

  // High within-group variance → aleatoric (same conditions, different outcomes)
  // High between-group variance → epistemic (different conditions, could learn which works)
  const aleatoricFraction = groupCount > 0 ? withinGroupVariance / groupCount / Math.max(0.01, totalVariance) : 0.5;

  return {
    epistemic: clamp(1 - aleatoricFraction, 0, 1) * (relevantOutcomes.length < 10 ? 0.8 : 1.0),
    aleatoric: clamp(aleatoricFraction, 0, 1),
    confidence: Math.min(1, relevantOutcomes.length / 20),
  };
}
```

**Effect**: Topics with high aleatoric uncertainty are **deprioritized** for exploration. "Allen sometimes replies and sometimes doesn't" (aleatoric) is less worth investigating than "I don't know if this research approach works" (epistemic).

#### 7.2 Learning Need Score (replaces simple EIG)

```typescript
function computeLearningNeed(
  node: ConceptNode,
  graph: ConceptGraph,
  uncertainty: UncertaintyDecomposition,
  worldModel: WorldModel,
): number {
  // Core: epistemic uncertainty × importance
  const epi = uncertainty.epistemic;
  const importance = Math.max(
    node.drive,                    // goal relevance
    node.salience * 0.5,           // novelty/urgency
    node.uncertainty * 0.3,        // raw uncertainty
  );

  // Connectivity bonus: learning propagates through well-connected nodes
  const neighborCount = graph.edges.filter(e => e.source === node.id || e.target === node.id).length;
  const connectivityBonus = Math.log1p(neighborCount) / Math.log1p(20);

  // Staleness: haven't explored recently → more to learn
  const daysSinceExplored = (Date.now() - (node.lastExternalBoostAt || node.lastActivated)) / 86400000;
  const stalenessBonus = Math.min(1.5, 1.0 + daysSinceExplored / 7);

  // Prediction error track record (from CS1)
  const predErrorBonus = avgPredictionErrorForConcept(node.id) > 0.4 ? 1.3 : 1.0;

  // World model epistemic error for this topic
  const wmEpi = worldModel.getEpistemicError(node.id);  // from transition stats confidence
  const wmBonus = wmEpi > 0.5 ? 1.2 : 1.0;

  return epi * importance * connectivityBonus * stalenessBonus * predErrorBonus * wmBonus;
}
```

#### 7.3 Value of Information (VOI): Curiosity Meets Planning

The key upgrade: exploration is **evaluated by its impact on future decisions**, not just information gain.

```typescript
function computeVOI(
  explorationAction: { nodeId: string, queryType: "search" | "recall_memory" | "ask_question" },
  currentBelief: BeliefState,
  planner: Planner,
  worldModel: WorldModel,
): number {
  // VOI = E[EU_after_exploration] - EU_before - cost

  // Step 1: Current EU (no exploration)
  const currentEU = planner.evaluateCurrentPlan(currentBelief);

  // Step 2: Predicted EU after exploration
  // Assume exploration reduces uncertainty on target node by ~30%
  const hypotheticalBelief = structuredClone(currentBelief);
  // Reduce epistemic uncertainty on target
  hypotheticalBelief.latent.topicViability = Math.max(0.3, currentBelief.latent.topicViability);
  // Simulate: how would the planner's best plan change with less uncertainty?
  const hypotheticalEU = planner.evaluateCurrentPlan(hypotheticalBelief);

  // Step 3: Cost of exploration
  const cost = explorationAction.queryType === "search" ? 0.15
    : explorationAction.queryType === "recall_memory" ? 0.02
    : explorationAction.queryType === "ask_question" ? 0.25;

  return hypotheticalEU - currentEU - cost;
}
```

**VOI interpretation**: If VOI > 0, exploring this topic would improve our best plan's expected utility enough to justify the cost. If VOI ≤ 0, exploring wouldn't change our decisions — skip it.

**Curiosity selection** (runs every slow-loop tick):
1. For each node with `U > 0.2`: compute `LearningNeed`
2. For top-5 by LearningNeed: compute `VOI` (more expensive, uses planner rollout)
3. For each candidate: compute `SelfCost` based on exploration action type and current `SelfState` (see CS5b.7)
4. **Final ranking**: `curiosityScore = 0.5 × LearningNeed + 0.3 × max(0, VOI) - 0.2 × SelfCost`
   - When VOI computation unavailable (planner cold start): fall back to `0.7 × LearningNeed - 0.3 × SelfCost`
   - SelfCost is **action-type-specific**: `recall_memory` costs almost nothing when fatigued, `ask_question` costs a lot when social_energy is low
5. Top 3 → `curiosityTargets`

This binds curiosity to planning AND self-awareness: **explore what helps you make better decisions**, using the cheapest viable exploration strategy given your current state. When fatigued, curiosity naturally shifts from `search` (expensive) to `recall_memory` (cheap). When social_energy is high, `ask_question` becomes viable.

### Learning Loop Architecture

```
                    ┌─────────────────────────────────┐
                    │     Concept Graph (dynamics)     │
                    │  spread → decay → compete → win  │
                    └──────────┬──────────────────────┘
                               │ winner cluster
                    ┌──────────▼──────────────────────┐
                    │  Gates (thought/reflect/act)      │
                    │  + Self Gate (CS5b)               │
                    │  → micro-thought / action pref    │
                    └──────────┬──────────────────────┘
                               │ action preference
    ┌──────────────────────────┤
    │                          │
    │  ┌───────────────────┐   │  ┌─────────────────────────┐
    │  │ Curiosity Engine  │   │  │   Planner               │
    │  │ E_epi/E_alea      │   │  │ beam search (K=3, D=3)  │
    │  │ LearningNeed      │   │  │ world model rollout     │
    │  │ VOI (via planner) │◄──┤  │ EU = world + self - cost│
    │  │ - SelfCost        │   │  │ selfModel.transition()  │
    │  └────────┬──────────┘   │  └────────┬────────────────┘
    │           │              │           │
    │           │ explore      │           │ plan bias
    │           │ targets      │           │
    │           │  ┌───────────▼───────────▼──────────┐
    │           │  │     Heartbeat (action select)     │
    │           └──│  reach_out / explore / post / ... │
    │              └──────────┬────────────────────────┘
    │                         │ action
    │              ┌──────────▼────────────────────────┐
    │              │     World (observable outcome)     │
    │              │  reply? discovery? progress?       │
    │              └──────────┬────────────────────────┘
    │                         │ outcome signal
    │  ┌──────────────────────▼────────────────────────┐
    │  │  OutcomeTracker + World Model + Self Model     │
    │  │  outcome → belief state + self state update    │
    │  │  (latent vars + conjugate priors + E_epi/alea  │
    │  │   + selfOutcomeUpdate + action family EWMA)    │
    │  └──────────┬────────────────────────────────────┘
    │             │ learned structure + updated beliefs + updated self
    └─────────────┘ back to Concept Graph ↑ (closed loop)
```

The key insight: **learning happens at three levels**, not the LLM level. The graph learns structure (edges, grounding, hierarchy). The world model learns environment dynamics (latent states, transition probabilities). The **self-model learns self-dynamics** (action costs, recovery rates, success probabilities). The curiosity engine learns what's worth exploring (epistemic vs aleatoric, self-cost-aware). The planner learns what works (action utilities via EU, including self-return). The LLM remains a verbalizer — it doesn't learn, but the substrate it verbalizes from does.

The full cognitive loop is now:
```
graph dynamics → thought → plan (beam search over world model + self model)
  → curiosity (VOI - SelfCost: explore what improves decisions at acceptable cost)
  → action → outcome
  → world model update (belief + latent + priors)
  → self model update (self state + action family table + self_efficacy)
  → graph update (edges + grounding + suppression)
  → better thoughts → better plans → better self-awareness
```

## Grounding Consistency Check

When grounding is old or anchor is weak, verify the reference still holds:

**Trigger**: `primaryRef.age > 14 days` AND (`anchor = "inferred"` OR `anchor = "speculative"`) AND cluster involves action/relationship/commitment concepts.

**Check** (rule-based, no LLM):
1. **Temporal relevance**: Is the memory about a time-bound event (keywords: "周末", "下周", "明天", specific dates)? If yes and event has passed → downgrade to `speculative`, block act gate.
2. **Entity state consistency**: If memory mentions a person + activity, check: is that person still in active context (recent conversation/pending interaction)? If not → add uncertainty marker.
3. **Contradiction check**: Scan recent memories for same key prefix. If newer memory exists that contradicts → use newer, mark old as `deprecated`.

**Output enforcement**: When consistency check flags an issue:
- Micro-thought text must include hedging ("好像之前...", "不确定现在还...")
- Act gate: blocked unless sanity check LLM explicitly confirms relevance
- Log: `groundingConsistencyCheck: { triggered: true, result: "stale" | "contradicted" | "ok" }`

**Knowledge TTL**: Certain node types carry a `ttlDays` field:
- News/prices/versions: `ttlDays = 7` (auto-expire, trigger re-verification intent)
- Personal facts (birthday, location): `ttlDays = null` (permanent)
- Plans/schedules: `ttlDays = 3` (expire quickly)
- On TTL expiry: anchor auto-downgrades to `inferred`, node salience boosted slightly (triggers re-investigation via curiosity)

## Edge Learning Safety Valve (CS2 extension)

### Negative Evidence (反证 weakening)

Current CS2 only has "pattern sparse → revert". Add: **counter-evidence actively weakens edges**:

```
for each edge (A, B):
  // existing: forward/backward temporal correlation
  // NEW: negative evidence
  count_A_without_B = times A activated but B did NOT follow within 90s
  if count_A_without_B > 5 AND edge.weight > 0.3:
    edge.weight -= 0.005 × (count_A_without_B - 5)
    // Stronger weakening when there's lots of counter-evidence
```

This implements predictive coding update: "A used to predict B, but B stopped following → weaken A→B". Prevents the graph from turning "偶然共现" into permanent "因果信念".

### Concept Semantic Anchor

Each node carries a stable `anchorText` (set once at creation, never updated by drift):

```typescript
anchorText: string;  // e.g., "量子计算：用量子力学原理处理信息的计算方式" (definition sentence)
```

- Set by `findOrCreateNode()` from the source text (memory excerpt, goal description, discovery query)
- Used as the **reference point** for merge/drift detection: if node's `termVector` Jaccard distance from `tokenize(anchorText)` exceeds 0.5 → "semantic drift alert" in daily report
- Merge scan uses `anchorText` similarity as a more stable signal than evolving `termVector`

## Auto-Calibration (online, data-driven)

CSI's bell-curve centers (novelty=0.5, rotation=9/h) are initial guesses. After sufficient data:

**Mechanism** (runs during wake-up consolidation, weekly):
1. Compute from last 7 days of `metrics.jsonl`:
   - `noveltyMedian` = median of all `noveltyScore` values during Green mode
   - `rotationMedian` = median of `winnerRotationRate` during Green mode
   - `entropyMedian` = median of `entropy` during Green mode
2. Update CSI bell-curve centers:
   - `N` center: `noveltyMedian` (instead of fixed 0.5)
   - `Rh` center: `rotationMedian` (instead of fixed 9)
   - `Eh` target: `entropyMedian` (instead of fixed 2.5)

   ```typescript

   ```
4. Decision log records `calibrationVersion` so metrics can be compared across calibration epochs

**Guard**: Only calibrate when `sampleSize > 500` slow-loop ticks (~8h of Green-mode data). If insufficient data → keep defaults.

## Multi-Scale Memory Protocol

Explicit time-layered memory pipeline with promotion/demotion rules:

| Layer | Timescale | Storage | Capacity | Promotion trigger | Demotion trigger |
|-------|-----------|---------|----------|-------------------|------------------|
| **STM** (short-term) | Minutes | `thoughtHistory` ring buffer | 30 entries | Concept in ≥3 thoughts over 2h → promote to MTM | FIFO eviction |
| **MTM** (medium-term) | Days | `insights.brainstem.MMDD` memories | ~30 per week | Insight cited in ≥3 reflections or conversations → promote to LTM | No citation for 30 days → archive |
| **LTM** (long-term) | Months | `opinions`, `knowledge`, `emotional` memories | Unlimited (existing memory store) | Manual or reflection-confirmed | Explicit deprecation only |

**Promotion gate**: STM→MTM promotion requires **evidence threshold**: the concept must appear in micro-thoughts with `anchor = grounded` at least twice. No speculative-only concepts get promoted.

**LTM update**: When an MTM insight contradicts existing LTM, trigger a reflection with: "你之前觉得X，但最近发现Y。要更新看法吗？" The reflection must produce a structured `memory_set` call to update — never silently overwrite.
